Text Classification — Overview

1.1 What is Text Classification?

Text classification (also called text categorization) is the task of automatically assigning predefined categories to text documents based on their content. It is the most fundamental and widely applied NLP task, forming the backbone of spam detection, sentiment analysis, topic labeling, intent recognition, and content moderation.

Text Classification in One Line

Given a text document d and a finite set of classes C = {c₁, c₂, ..., cₙ}, assign the most appropriate class c* ∈ C to d. The assignment is learned from labeled training examples — not hand-coded rules.

1.2 Text Classification Tasks — Examples

Spam Detection

Email → {Spam, Not-Spam}. One of the earliest and most successful commercial NLP applications. Naïve Bayes became famous solving this task.

Sentiment Analysis

Review → {Positive, Negative, Neutral}. Applied to product reviews, tweets, news articles, customer feedback. Detailed in Part 4.

Topic Classification

News article → {Sports, Politics, Technology, Business, Science}. Enables automatic content routing and recommendation.

Intent Detection

User utterance → {BookFlight, CancelOrder, GetHelp, CheckStatus}. Core of every chatbot and virtual assistant's NLU layer.

Language Identification

Text → {English, Hindi, French, German, ...}. Preprocessing step for multilingual systems. Character n-gram features work well.

Content Moderation

Post → {Safe, Hate Speech, Harassment, NSFW}. Critical for social media platforms at scale. Multi-label classification common.

Machine Learning for Text — The Pipeline

2.1 Why ML for Text Classification?

  • ▸
    Scale: Millions of emails, tweets, reviews, and documents cannot be manually classified. ML models classify at millions-per-second speeds once trained.
  • ▸
    Adaptability: Spammers adapt their language — an ML model can be retrained on new data. Hand-coded rules quickly go stale.
  • ▸
    Generalization: ML models learn patterns from labeled examples and generalize to new, unseen text. They handle paraphrases, synonyms, and novel sentence structures that rule-based systems miss.
  • ▸
    Language agnosticism: The same ML pipeline applies to any language given labeled training data — just change the preprocessing and vocabulary.

Words and Term Frequency

3.1 Words as Features — The Fundamental Idea

To apply any ML algorithm to text, we must convert variable-length text strings into fixed-length numeric vectors. The most natural approach: use the words (terms) in the text as features. Each unique word in the vocabulary becomes one dimension. This treats text as a "bag" — ignoring grammar and word order, keeping only what words appear and how often.

The Bag-of-Words Assumption

"A document can be represented as an unordered set of its words, where each word's frequency carries meaning about the document's topic and sentiment." Two documents sharing many words on "machine learning" are likely about the same topic, even if sentence structure differs. The bag-of-words model ignores "not good" vs "good not" — a limitation addressed by n-grams and deeper models.

3.2 Term Frequency (TF)

Term Frequency TF(t, d) counts how many times term t appears in document d. It captures how central a word is to the document's content.

Term Frequency — Three Variants
Raw count: TF(t, d) = count of term t in document d
← simple count. Most common form.

Relative frequency: TF(t, d) = count(t, d) / Σₜ' count(t', d)
← normalize by total words in d. Documents of any length comparable.

Log normalization: TF(t, d) = 1 + log(count(t, d)) if count(t,d) > 0, else 0
← reduces impact of very high-frequency words. Diminishing returns.

Binary: TF(t, d) = 1 if t appears in d, else 0
← presence/absence only. Ignores frequency. Used in some Naïve Bayes variants.

3.3 Vocabulary Construction

  1. Tokenize all documents into words.
  2. Normalize: lowercase, remove punctuation, optionally remove stopwords, stem or lemmatize.
  3. Collect all unique terms across all training documents → this is the vocabulary V.
  4. Assign index to each term: V = {term₀:0, term₁:1, ..., term_{|V|-1}:|V|-1}.
  5. Represent each document as a vector of length |V| where entry j = TF(termⱼ, doc).

Bag-of-Words and CountVectorizer

4.1 Bag-of-Words (BoW) Matrix

The Bag-of-Words model represents the entire corpus as a matrix where each row is a document and each column is a vocabulary term. The cell at (row i, column j) contains TF(termⱼ, docᵢ) — how often word j appears in document i.

Document catdoglogmatpetsat
Doc 1: "the cat sat on the mat" 100101
Doc 2: "the dog sat on the log" 011001
Doc 3: "cats and dogs are pets" 110010
  • ▸
    Sparsity: Real BoW matrices are extremely sparse. A corpus of 50,000 documents and vocabulary of 100,000 terms → 5 billion cells, but only ~0.01–0.1% are non-zero. Sparse matrix formats (CSR, CSC) are essential for memory efficiency.
  • ▸
    High dimensionality: The vocabulary size becomes the feature space dimensionality. Large corpora have 50,000–500,000 unique terms after preprocessing. This is the "curse of dimensionality" for text — handled by vocabulary limits, min_df, and max_df filters.
  • ▸
    Word order lost: "The cat bit the dog" and "The dog bit the cat" have identical BoW representations. Bigrams and trigrams partially address this by treating word pairs/triples as features.

4.2 CountVectorizer — Scikit-learn Implementation

CountVectorizer from scikit-learn is the standard tool for converting a collection of text documents to a matrix of token counts (raw TF values). It handles tokenization, lowercasing, stopword removal, vocabulary building, and matrix construction in one step.

Python — CountVectorizer Deep Dive
from sklearn.feature_extraction.text import CountVectorizer import pandas as pd corpus = [ "I love this product it is amazing", "This product is terrible I hate it", "Amazing quality highly recommended", "Worst purchase ever do not buy", ] # Basic CountVectorizer cv = CountVectorizer( lowercase=True, # convert to lowercase stop_words='english', # remove English stopwords min_df=1, # min doc frequency (ignore rarer terms) max_df=0.95, # max doc freq (ignore terms in 95%+ docs) max_features=5000, # limit vocabulary size ngram_range=(1,2), # include unigrams AND bigrams token_pattern=r'\b[a-z]{2,}\b' # only words 2+ chars ) X = cv.fit_transform(corpus) # sparse matrix (4 docs × vocab_size) print("Vocabulary:", cv.get_feature_names_out()) print("Matrix shape:", X.shape) print("Dense matrix:\n", X.toarray()) # View as DataFrame df = pd.DataFrame(X.toarray(), columns=cv.get_feature_names_out()) print(df) # CORRECT: fit on train, transform both cv_prod = CountVectorizer() X_train_cv = cv_prod.fit_transform(train_docs) # learn vocab from train X_test_cv = cv_prod.transform(test_docs) # apply same vocab to test

4.4 Limitations of Pure Count Vectors

  • ▸
    Dominated by frequent words: "the", "a", "is", "of" appear in every document. Even after stopword removal, domain-specific frequent words (in medical text: "patient", "treatment") dominate raw counts and dilute discriminative signal.
  • ▸
    No importance weighting: Raw counts treat all words equally. "machine" appearing twice is twice as important as "learning" appearing once — but neither's frequency reflects its discriminative power for the topic.
  • ▸
    Document length bias: Longer documents naturally have higher counts for all terms. A short positive review with "excellent" once and a long lukewarm review with "excellent" twice might rank equally — unfairly.
  • ▸
    Solution — TF-IDF: Weights each term by how informative it is across the corpus — high weight for terms that are frequent in a specific document but rare across documents. Solves all three problems above.
Section 5

Inverse Document Frequency (IDF)

5.1 The IDF Intuition

A word that appears in almost every document (like "the" or "is") carries no information about what makes any specific document unique — it can't help distinguish a sports article from a finance article. Conversely, a word like "photosynthesis" appearing in only a handful of biology documents tells us a lot about those documents' content. IDF measures this rarity signal.

Inverse Document Frequency (IDF) — Variants
N = total number of documents in the corpus
df(t) = number of documents containing term t

Standard IDF: IDF(t) = log( N / df(t) )
Smoothed IDF: IDF(t) = log( N / (df(t) + 1) ) + 1 ← sklearn default
sklearn IDF: IDF(t) = log( (N+1) / (df(t)+1) ) + 1 ← add-one smoothing

Interpretation:
Term in ALL docs: df(t) = N → IDF = log(1) = 0 ← zero weight, useless
Term in 1 doc: df(t) = 1 → IDF = log(N) ← maximum weight, very distinctive
Term in 10% docs: df(t) = N/10 → IDF = log(10) ≈ 2.3

Smoothing avoids zero-IDF for terms appearing in all documents and zero-division for terms not in training vocabulary.

TF-IDF — Full Derivation and TfidfVectorizer

6.1 TF-IDF — The Combined Weight

TF-IDF (Term Frequency — Inverse Document Frequency) multiplies the term frequency (how often the word appears in the document) by the inverse document frequency (how rare the word is across the corpus). This gives high weight to words that are frequent in a specific document but rare elsewhere — the most informative words for distinguishing documents.

TF-IDF — Complete Formula (sklearn default)
TF-IDF(t, d, D) = TF(t, d) × IDF(t, D)

TF(t, d) = count(t in d) / total_words_in_d ← relative frequency
IDF(t, D) = log( (N+1)/(df(t)+1) ) + 1 ← smoothed IDF

N = |D| = total documents in corpus
df(t) = number of docs in D containing term t

After computing TF-IDF matrix, each row is L2-normalized:
v_normalized = v / ‖v‖₂ ← divide by Euclidean norm
← ensures documents of different lengths have comparable vectors

High TF-IDF: word appears OFTEN in THIS doc but RARELY across all docs
→ Highly discriminative for this document's topic

Low TF-IDF: word appears in MANY docs OR rarely in this doc
→ Not useful for distinguishing this document

6.2 TF-IDF — Full Worked Example

TF-IDF Calculation — "machine learning deep learning" in 4-document corpus
Corpus (N=4)
D1:"machine learning deep learning" | D2:"machine learning" | D3:"deep neural networks" | D4:"learning from data"
Focus: Doc D1

TF("learning", D1)

Key insight
"learning" appears more in D1 but has lower TF-IDF because it's more common across the corpus. TF-IDF correctly penalizes common words.

6.3 TfidfVectorizer — Scikit-learn Implementation

Python — TfidfVectorizer with Full Options
from sklearn.feature_extraction.text import TfidfVectorizer from sklearn.linear_model import LogisticRegression from sklearn.pipeline import Pipeline train_docs = ["I love this movie it was amazing", "Terrible film I hated every minute", "Outstanding performance highly recommend", "Boring and predictable waste of time"] train_labels = [1, 0, 1, 0] # 1=positive, 0=negative tfidf = TfidfVectorizer( ngram_range=(1, 2), # unigrams + bigrams sublinear_tf=True, # use log(TF)+1 instead of raw TF min_df=1, max_df=0.9, # ignore terms in 90%+ docs norm='l2', # L2-normalize each document vector max_features=10000, stop_words='english' ) # Build complete pipeline: vectorize → classify pipeline = Pipeline([ ('tfidf', tfidf), ('classifier', LogisticRegression(C=1.0)) ]) pipeline.fit(train_docs, train_labels) # Predict new document new_doc = ["This film was absolutely wonderful"] print(pipeline.predict(new_doc)) # → [1] (positive) print(pipeline.predict_proba(new_doc)) # → [[0.12, 0.88]] # Inspect top TF-IDF features feature_names = tfidf.get_feature_names_out() X = tfidf.transform(train_docs) print("Top features for Doc 1:", sorted(zip(X[0].toarray()[0], feature_names), reverse=True)[:5])

7.2 N-gram Features for Text Classification

  • ▸
    Why n-grams matter for classification: "not good" is negative but "good" is positive — unigrams miss the negation. "highly recommend" strongly signals positive sentiment as a bigram. "customer service" is a product feature. Bigrams capture local context that dramatically improves accuracy.
  • ▸
    Typical setting: ngram_range=(1,2) in TfidfVectorizer adds all unigrams and bigrams. For spam detection: (1,3) including trigrams often helps capture phishing phrases like "click here now."
  • ▸
    Dimensionality explosion: Adding bigrams can multiply vocabulary size by 5–20×. Use max_features to limit. For bigrams: typically keep top 50,000–100,000 features after TF-IDF weighting selects most informative ones.

Naïve Bayes — Probabilistic Foundation

8.1 Bayes' Theorem — The Core

Naïve Bayes is a probabilistic classifier grounded in Bayes' theorem. It computes the posterior probability of each class given the observed features, then picks the class with the highest probability.

Bayes' Theorem
P(c | d) = P(d | c) · P(c) / P(d)

P(c | d) = posterior — probability of class c given document d
P(d | c) = likelihood — probability of seeing document d if class is c
P(c) = prior — baseline probability of class c (from training set)
P(d) = evidence — probability of document d (same for all classes)

For classification, we want:
c* = argmax_c P(c | d) = argmax_c P(d | c) · P(c)
← P(d) is a constant → can be ignored for argmax

8.2 The Naïve Assumption — Conditional Independence

The Naïve Bayes Assumption

Given the class label c, all features (words) are conditionally independent of each other:

P(w₁, w₂, ..., wₙ | c) = P(w₁|c) · P(w₂|c) · ... · P(wₙ|c) = Πᵢ P(wᵢ|c)

This is "naïve" because it's clearly not true — words co-occur ("machine" and "learning" appear together). But despite this unrealistic assumption, Naïve Bayes achieves surprisingly good performance in practice, especially for text classification where feature independence is a reasonable approximation.

8.3 Naïve Bayes Classifier — Full Derivation

Naïve Bayes for Text Classification — Step by Step
Document d = (w₁, w₂, ..., wₙ) ← sequence of n words
Classes: C = {c₁, c₂, ..., cₖ}

Posterior by Bayes' theorem:
P(c | d) = P(d | c) · P(c) / P(d)

Apply naïve independence:
P(d | c) = Πᵢ P(wᵢ | c)

Classification rule:
c* = argmax_c P(c) · Πᵢ P(wᵢ | c)

In log space (avoids numerical underflow for long documents):
c* = argmax_c [ log P(c) + Σᵢ log P(wᵢ | c) ]

Learning (estimating parameters from training data):
P(c) = count(docs in class c) / count(total docs) ← prior
P(w|c) = count(w in class c docs) / count(all words in class c docs) ← likelihood

8.4 Why Log Space? — Numerical Underflow

  • ▸
    The problem: P(w|c) for a single word is typically 0.001–0.00001. For a document of 100 words: Π P(wᵢ|c) ≈ 0.001^100 = 10^{-300} → underflows to 0.0 in floating point. We can't compare 0.0 to 0.0.
  • ▸
    Log transformation: log(Π P(wᵢ|c)) = Σ log P(wᵢ|c). Log of probabilities is in range (−∞, 0). For 100 words: Σ log(0.001) = 100 × (−6.9) = −690. Perfectly representable as a float. argmax of logs = argmax of originals (log is monotone).

8.5 Laplace Smoothing — Handling Zero Probabilities

Laplace (Add-1) Smoothing for Naïve Bayes
Problem: A word not seen in training for class c gives P(w|c)=0
→ Entire product P(d|c) = 0 for any doc containing this word
→ Posterior P(c|d) = 0 regardless of other evidence. WRONG!

Laplace (Add-1) smoothed estimate:
P(w | c) = (count(w, c) + 1) / (count(all words in c) + |V|)

count(w, c) = times word w appeared in class c documents
count(all, c) = total word tokens in class c documents
|V| = vocabulary size (normalizes so probabilities sum to 1)

Add-k smoothing (more general):
P(w | c) = (count(w, c) + k) / (count(all, c) + k·|V|)
k = 0.1–1.0; k<1 adds less smoothing (useful when k=1 over-smooths)

Naïve Bayes Variants

10.1 Three Main Variants

Variant 1 — Most Common for NLP
Multinomial Naïve Bayes

Input features: Word count vectors (CountVectorizer output). Works with non-negative integer or float counts.

Likelihood model: P(d|c) = multinomial distribution over word counts. Models the probability of generating a document with exactly the observed word frequencies from class c's word distribution.

Best for: Text classification with bag-of-words features. Spam detection, topic classification, sentiment analysis. The standard choice for text.

sklearn: MultinomialNB(alpha=1.0). Use with CountVectorizer. alpha = Laplace smoothing parameter.
Variant 2 — Binary Features
Bernoulli Naïve Bayes

Input features: Binary word presence/absence vectors (1 if word in doc, 0 otherwise). Ignores frequency — only whether a word appears at all.

Likelihood model: P(d|c) = Πᵢ [P(wᵢ|c)^bᵢ · (1−P(wᵢ|c))^(1−bᵢ)] where bᵢ = 1 if word wᵢ appears in d. Explicitly penalizes absence of words — missing a spam word counts as evidence against spam.

Best for: Short documents (tweets, titles) where frequency is less informative than presence. Binary text features.

sklearn: BernoulliNB(). Use with CountVectorizer(binary=True) to convert counts to 0/1.
Variant 3 — Continuous Features
Gaussian Naïve Bayes

Input features: Continuous-valued features. Assumes features follow a Gaussian (normal) distribution within each class.

Likelihood model: P(xᵢ|c) = N(xᵢ; μᵢc, σᵢc²) — Gaussian with class-specific mean and variance per feature. Parameters (μᵢc, σᵢc²) estimated from training data.

Best for: Non-text features — continuous measurements (temperature, height, sensor readings). Less common for NLP but used when features are TF-IDF float vectors.

sklearn: GaussianNB(). Does not use Laplace smoothing (variance takes care of zero-probability issues).
Variant 4 — Modern NLP
Complement Naïve Bayes (CNB)

Idea: Addresses Multinomial NB's bias toward majority class by training each class classifier using ALL OTHER classes' data as the "complement." Particularly useful for imbalanced datasets.

How it works: P(w|c̄) = probability of word w across all classes EXCEPT c. The class whose complement least explains the document is chosen.

Best for: Imbalanced text classification where MultinomialNB over-predicts the majority class. Often outperforms MultinomialNB on text benchmarks.

sklearn: ComplementNB(alpha=1.0). Usually better than MultinomialNB for unbalanced text classification.

10.2 Naïve Bayes — Advantages and Disadvantages

AdvantagesDisadvantages
Extremely fast: O(|V| × k) training — just count word frequencies per class. No iterative optimization. Classifies millions of documents per second. Naïve independence assumption: Words are never truly independent — "machine" and "learning" co-occur. Violating this means probability estimates are often wrong (but classification decisions can still be correct).
Works well with little data: Estimates are stable even with small training sets because parameters are simple counts, not gradient-descent solutions. Poor probability calibration: Posterior probabilities are systematically over-confident (0.999 when true probability is 0.70). Do not trust the magnitude of NB's probabilities — only the argmax decision.
Handles high dimensions: No issues with |V| = 100,000 features. Linear in vocabulary size. Memory-efficient. Cannot learn feature interactions: "not good" = {not: negative, good: positive} → wrongly predicted positive. Feature independence prevents capturing negation, irony, and complex patterns.
Robust to irrelevant features: Each feature's weight is estimated independently. Noisy/irrelevant features slightly hurt but rarely catastrophically. Sensitive to feature correlations: If two correlated features are both present, NB double-counts their evidence, leading to over-confident predictions.
Online learning: New training examples update only the relevant count tables — no need to retrain from scratch. Continuous features: Multinomial NB requires non-negative counts. Gaussian NB for continuous features assumes normality which is often violated.

Sentiment Analysis — Overview and Levels

11.1 What is Sentiment Analysis?

Sentiment Analysis (also called Opinion Mining) is the computational study of opinions, sentiments, attitudes, and emotions expressed in text. It automates the task of determining whether text expresses a positive, negative, or neutral attitude toward a subject — at any level of granularity.

Business Motivation

A company launching a new product receives 50,000 reviews in the first week. Manually reading all of them would take weeks. Sentiment analysis extracts the overall sentiment, identifies specific product aspects that customers love or hate, tracks how sentiment changes over time, and benchmarks against competitors — in seconds. This "voice of the customer" at scale drives product decisions, marketing, and customer service strategy.

11.2 Levels of Sentiment Analysis

LevelGranularityExampleOutput
Document-level Entire review/document "I love this camera. The battery life is excellent and photos are stunning." Positive (overall)
Sentence-level Individual sentence S1: "The food was delicious." → Positive. S2: "The service was slow." → Negative. Per-sentence polarity
Aspect-level (ABSA) Specific aspects/features "The food was great but service was terrible and prices were fair." food:+, service:−, price:neutral
Entity-level Named entities "Apple released a great iPhone but Samsung's new model is better." Apple:mixed, Samsung:positive
Emotion detection Specific emotion categories "I am so excited but also anxious about the exam." {joy: 0.7, fear: 0.5, ...}

11.3 Challenges in Sentiment Analysis

  • ▸
    Negation: "This is NOT a good product" — the negation completely reverses "good"'s polarity. Simple BoW misses this. N-gram features help ("not good" as a bigram). Dependency parsing properly handles negation scope.
  • ▸
    Sarcasm and irony: "Oh great, another software update that breaks everything!" — positive words, negative sentiment. Extremely hard — requires pragmatic understanding and world knowledge. State-of-the-art models still struggle (~70–75% accuracy).
  • ▸
    Domain shift: "This laptop is sick!" — "sick" means excellent in tech slang, terrible in medical contexts. Sentiment lexicons from one domain may be misleading in another. Domain-specific training data and lexicons needed.
  • ▸
    Comparative sentences: "This phone is better than my old one" — positive about new phone, implies negative about the old. Standard sentiment classification doesn't capture comparisons.
  • ▸
    Subjectivity detection: Before classifying sentiment, must determine if the text is even subjective (opinionated) or objective (factual). "The camera has 12MP resolution" is factual, not sentimental. Subjectivity classification precedes sentiment classification.
Section 12

Sentiment Analysis — Methods

12.1 Three Approaches to Sentiment Analysis

Approach 1 — Lexicon-Based
Dictionary / Lexicon Methods

Idea: Maintain a dictionary of words pre-labeled with their sentiment polarity and intensity. Score a document by summing the sentiment values of its words.

Popular lexicons: VADER (Valence Aware Dictionary for sEntiment Reasoning) — specifically tuned for social media. AFINN — 2477 words scored −5 to +5. SentiWordNet — WordNet synsets with positive/negative scores. Opinion Lexicon (Liu) — 6800 positive and negative words.

VADER handling: Capitalisation (GREAT vs great), punctuation (!!! amplifies), degree modifiers (very good), negation (not good). Specifically designed for short social media text.

Pros: No training data needed, interpretable. Cons: Misses domain-specific slang, sarcasm; lexicons go stale.
Approach 2 — Classical ML
ML-Based Sentiment Classification

Pipeline: Labeled training data → TF-IDF features (with bigrams) → Naïve Bayes / Logistic Regression / SVM classifier. Same pipeline as general text classification.

Key feature additions for sentiment: Negation handling (append _NEG suffix to words after negation: "not good" → "not good_NEG"). Intensifier detection ("very", "extremely"). Punctuation features (! count, ALL-CAPS ratio). Emoji conversion to text.

Typical accuracy: Naïve Bayes: 82–87%. Logistic Regression + TF-IDF bigrams: 88–92% on standard benchmarks. LinearSVM: 89–93%.

Strong baseline. Requires labeled data. Fast training and inference. Often sufficient for production.
Approach 3 — Deep Learning
Neural and Transformer Methods

Progression: CNN sentence classifier (Kim 2014) → BiLSTM with attention → BERT fine-tuned → sentiment-specific pre-trained models (SentimentBERT, RoBERTa-fine-tuned).

BERT for sentiment: Fine-tune BERT on labeled sentiment data. [CLS] token output → Dense(2) + Softmax. 3–5 epochs. Achieves 93–96% on SST-2 (Stanford Sentiment Treebank). Captures context, negation, complex expressions that BoW misses.

Aspect-based: Token-level BERT + aspect extraction + relation classification between aspect and sentiment span.

Best accuracy. Requires GPU. Slow inference. Worth it for high-stakes applications (brand monitoring, financial sentiment).

Section 14

Evaluation Metrics for Text Classification

14.1 Complete Metrics Reference

All Key Classification Metrics
Accuracy = (TP + TN) / (TP + TN + FP + FN)
← proportion correct. Unreliable for imbalanced classes.

Precision (P) = TP / (TP + FP)
← of all docs classified as class C, what fraction actually are C?

Recall (R) = TP / (TP + FN)
← of all docs actually in class C, what fraction did we correctly find?

F1-Score = 2 · P · R / (P + R) ← harmonic mean
Fβ-Score = (1+β²) · P · R / (β²·P + R) ← β>1 weights recall more

Multi-class averaging:
Macro = (1/k) Σ_c metric(c) ← equal weight per class
Micro = aggregate TP,FP,FN → metric ← weight by class frequency
Weighted = Σ_c support(c)/N × metric(c) ← weight by class count

AUC-ROC: area under ROC (TPR vs FPR curve). Range [0,1]. >0.9 = excellent.

14.3 Special Considerations for Text Evaluation

  • ▸
    Class imbalance is the norm: In spam: 5–20% spam, 80–95% ham. In sentiment: often fewer neutral examples. Always use macro-F1 or class-specific metrics alongside accuracy. Never report accuracy alone for text classification.
  • ▸
    Cross-validation for small datasets: NLP datasets are often small (hundreds to thousands of labeled examples per class). Use stratified k-fold CV to get reliable performance estimates rather than a single train/test split.
  • ▸
    Human baseline: For difficult text tasks (sarcasm detection, fine-grained emotion), humans achieve only 70–80% agreement. A model at 75% may be near-human performance even though it seems low.
  • ▸
    Error analysis is essential: Always manually inspect 50–100 misclassified examples. Common patterns: model misses negations, misclassifies domain slang, fails on very short texts, struggles with code-mixed text. These insights guide targeted improvements.

Exam Point — Naïve Bayes for Text: Key Facts

Why Naïve Bayes works for text despite the independence assumption: Even though words are not independent (co-occurrences are common), the argmax decision is often still correct. The ranking of class probabilities is often preserved even when the probability magnitudes are wrong. This is called "diagnostic accuracy without calibration."

Multinomial vs Bernoulli NB: Multinomial uses word counts (how many times); Bernoulli uses word presence/absence (yes/no). For long documents: Multinomial generally better. For short documents: Bernoulli competitive because frequency is less informative than presence.

CountVectorizer vs TfidfVectorizer with NB: MultinomialNB requires non-negative features — works directly with CountVectorizer (integer counts). TfidfVectorizer produces float features that are still non-negative so also works. For NB specifically, raw counts are often preferred over TF-IDF.

Module 3 — Complete Summary

  • 1.
    Text Classification: Automatically assign predefined categories to text. Types: binary (spam/ham), multi-class (topic), multi-label (tags), hierarchical. ML pipeline: raw text → clean → tokenize → vectorize → model → evaluate.
  • 2.
    Words & Term Frequency: TF(t,d) = count of term t in doc d. Variants: raw count, relative frequency (÷total words), log normalization (1+log(count)), binary (0/1). Vocabulary V = all unique terms in training corpus; each term → one feature dimension.
  • 3.
    Bag-of-Words: Represent each document as a vector of term frequencies, ignoring word order. Extremely sparse matrix (most cells = 0). CountVectorizer handles: tokenization, lowercasing, stopwords, vocabulary building, ngram generation (ngram_range), vocabulary limits (max_features, min_df, max_df).
  • 4.
    IDF: IDF(t) = log(N/df(t)). Zero for words in all documents. Maximum for words in only one document. sklearn: IDF = log((N+1)/(df+1))+1. Purpose: down-weight common uninformative words, up-weight distinctive rare words.
  • 5.
    TF-IDF: TF-IDF(t,d) = TF(t,d) × IDF(t). High weight: frequent in THIS document, rare across ALL documents → most informative signal for the document's topic. L2 normalization makes documents length-comparable. Use TfidfVectorizer with sublinear_tf=True for large texts.
  • 6.
    Naïve Bayes: c* = argmax_c [log P(c) + Σᵢ log P(wᵢ|c)]. Prior P(c) = class frequency. Likelihood P(w|c) = (count(w,c)+1)/(total_in_c+|V|) with Laplace smoothing. Log space prevents underflow. Naïve assumption: words conditionally independent given class.
  • 7.
    NB Variants: MultinomialNB (word counts, standard text); BernoulliNB (binary presence, short texts); GaussianNB (continuous features); ComplementNB (imbalanced datasets). MultinomialNB is the go-to for NLP classification.
  • 8.
    Sentiment Analysis: Levels: document → sentence → aspect-based → entity → emotion. Methods: (1) Lexicon-based (VADER, AFINN, SentiWordNet) — no training data needed. (2) ML (TF-IDF + NB/LR/SVM) — standard production approach. (3) BERT fine-tuned — best accuracy for complex sentiment.
  • 9.
    Confusion Matrix: TP=correct positive, TN=correct negative, FP=false alarm (Type I), FN=miss (Type II). Multi-class: k×k matrix. Diagonal = correct. Off-diagonal = confusions between specific classes.
  • 10.
    Metrics: Precision=TP/(TP+FP). Recall=TP/(TP+FN). F1=2PR/(P+R). Multi-class: macro (equal weight), weighted (by support), micro (aggregate). Use macro-F1 for imbalanced classes. Prefer Recall when FN costly (e.g., hate speech missed); Precision when FP costly (e.g., legitimate email blocked).
