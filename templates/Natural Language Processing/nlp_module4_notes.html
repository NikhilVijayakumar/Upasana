Language Modeling — N-gram Models

1.1 What is a Language Model?

A Language Model (LM) is a probability distribution over sequences of words. Given a sequence of words, it assigns a probability that this sequence is a plausible, grammatically valid, and meaningful sentence in a given language. Language models are the foundation of all modern NLP — from speech recognition to text generation to machine translation.

Language Model — Core Function

A language model answers two questions: (1) Scoring: How probable is sentence W = w₁w₂...wₙ? → P(W). Used in speech recognition to pick the most likely transcript. (2) Generation: Given a prefix w₁...wₖ, what is the most likely next word wₖ₊₁? → argmax_w P(w | w₁...wₖ). Used in autocomplete, text generation, machine translation decoding.

1.2 The Chain Rule of Probability

The joint probability of a sentence is decomposed into a product of conditional probabilities using the chain rule:

Chain Rule Decomposition
P(w₁, w₂, ..., wₙ) = P(w₁) · P(w₂|w₁) · P(w₃|w₁,w₂) · ... · P(wₙ|w₁,...,wₙ₋₁)

= Π_{i=1}^{n} P(wᵢ | w₁, w₂, ..., wᵢ₋₁)

Example: P("The cat sat on the mat")
= P(The) × P(cat|The) × P(sat|The cat) × P(on|The cat sat)
× P(the|The cat sat on) × P(mat|The cat sat on the)

Problem: P(wₙ|w₁,...,wₙ₋₁) requires seeing every possible prefix
→ Combinatorial explosion → impossible to estimate from data

1.3 The Markov Assumption — N-gram Models

To make language modeling tractable, we apply the Markov assumption: each word depends only on the previous N-1 words (not the entire history). This gives us the N-gram language model family:

N-gram Language Model — Probability Estimation
Unigram (N=1): P(wᵢ | w₁...wᵢ₋₁) ≈ P(wᵢ)
← word frequency only; ignores all context

Bigram (N=2): P(wᵢ | w₁...wᵢ₋₁) ≈ P(wᵢ | wᵢ₋₁)
← depends only on PREVIOUS word

Trigram (N=3): P(wᵢ | w₁...wᵢ₋₁) ≈ P(wᵢ | wᵢ₋₂, wᵢ₋₁)
← depends on previous TWO words

General N-gram: P(wᵢ | wᵢ₋ₙ₊₁, ..., wᵢ₋₁)

Maximum Likelihood Estimation (MLE) of bigram probability:
P(wᵢ | wᵢ₋₁) = C(wᵢ₋₁, wᵢ) / C(wᵢ₋₁)

C(wᵢ₋₁, wᵢ) = count of bigram (wᵢ₋₁, wᵢ) in training corpus
C(wᵢ₋₁) = count of unigram (wᵢ₋₁) in training corpus

1.5 Perplexity — Evaluating Language Models

Perplexity — The Standard LM Evaluation Metric
PP(W) = P(w₁w₂...wₙ)^{−1/N}
= ⁿ√(1 / P(w₁w₂...wₙ))
= 2^{H(W)} where H(W) = −(1/N)·Σ log₂ P(wᵢ|context)

Lower perplexity = better model (more confident, less "perplexed")

Interpretation: Average branching factor — how many words could
plausibly follow at each position according to the model.

Benchmarks on WSJ (Wall Street Journal) corpus:
Unigram LM: PP ≈ 962 ← very confused
Bigram LM: PP ≈ 170
Trigram LM: PP ≈ 74 ← better but still uncertain
Neural LM: PP < 40 ← near state-of-art
GPT-3: PP ≈ 20 ← excellent

Random model (V=50,000): PP = 50,000 (maximally confused)

Section 2

Neural Language Models

2.1 From N-gram to Neural Models

  • ▸
    Limitation of N-gram LMs: Sparsity — most N-grams never appear in training data. Fixed context window — trigrams only see 2 previous words. No generalization — "The cat sat" and "The feline rested" have no connection in a trigram model despite similar meaning.
  • ▸
    Feedforward Neural LM (Bengio 2003): Words are represented as dense vectors (embeddings). The neural network inputs the embeddings of the previous N-1 words and outputs a probability distribution over the vocabulary. Similar words get similar embeddings → generalization across synonyms.
  • ▸
    RNN Language Models: Replace the fixed N-gram window with a recurrent hidden state hₜ = f(hₜ₋₁, xₜ). Theoretically unlimited context (in practice, vanishing gradient limits effective range). LSTM/GRU extend this with gating mechanisms.
  • ▸
    Transformer Language Models: Self-attention over all previous tokens simultaneously. O(1) path between any two positions (vs O(n) for RNNs). Parallelizable. GPT, LLaMA use causal (left-to-right) attention for autoregressive LM. BERT uses bidirectional masked LM.

2.2 Language Modeling Applications

Text Generation

Sample from P(wₜ|w₁...wₜ₋₁) repeatedly to generate coherent text. Temperature controls randomness. GPT-4, Claude, LLaMA use this.

Speech Recognition

Acoustic model → multiple hypotheses → LM scores each → pick highest P(words)×P(acoustics|words). LM corrects acoustic errors using word-level constraints.

Machine Translation

Target LM score guides decoder — ensures output is fluent in the target language even when translation model is uncertain about word order.

Spell & Grammar Check

Low LM probability for a word in context → likely wrong. "I eated the apple" → P(eated|I) ≪ P(ate|I). Flags as error, suggests correction.

Autocomplete

P(w | previous words) → suggest most probable next word. Phone keyboard, Gmail Smart Compose, GitHub Copilot all use LM predictions.

Pretrained Representations

BERT, GPT learn rich word representations by predicting masked/next tokens. These representations transfer to all downstream NLP tasks.

Sequence Tagging — Concepts

3.1 What is Sequence Tagging?

Sequence tagging (also called sequence labeling) is the task of assigning a label to each token in an input sequence. Unlike document-level classification (which assigns one label to the whole input), sequence tagging assigns one label per token — recognizing local structure, boundaries, and roles within the text.

Sequence Tagging vs Text Classification

Text Classification: INPUT: "Apple is a great company." OUTPUT: one label → Positive (sentiment for the whole sentence).

Sequence Tagging: INPUT: "Apple is a great company." OUTPUT: one label per token → [B-ORG, O, O, O, O] (Apple = Organization entity).

Sequence tagging requires understanding not just what is in the text, but where it is and what role each token plays.

Section 4

Predicting Sequence of Tags

4.1 The Sequence Tagging Problem — Formal

Given an input sequence X = (x₁, x₂, ..., xₙ), find the output tag sequence Y = (y₁, y₂, ..., yₙ) that maximizes the joint probability P(Y|X). The critical insight is that tags are not independent — neighboring tags are strongly correlated (a DT is almost always followed by NN or JJ, never by VBD).

Sequence Tagging — Structured Prediction
Naive approach: y*ᵢ = argmax_y P(y | xᵢ) ← classify each token independently
← ignores dependencies between adjacent tags

Better approach (structured prediction):
Y* = argmax_Y P(Y | X) = argmax_Y P(y₁,...,yₙ | x₁,...,xₙ)

Factored using Markov assumption on tags:
P(Y|X) ≈ Π_{i=1}^{n} P(yᵢ | yᵢ₋₁, X)

This captures tag-to-tag transitions (label dependencies):
DT → JJ/NN (high probability)
DT → VBD (very low probability)
VBZ → NN/DT (sentence object follows)
IN → DT/NN (preposition takes noun phrase object)

4.2 Approaches to Sequence Tag Prediction

Classical Probabilistic
Hidden Markov Model (HMM)

Generative model with: Transition probability P(yᵢ|yᵢ₋₁) — how likely is tag yᵢ after yᵢ₋₁? Emission probability P(xᵢ|yᵢ) — how likely is word xᵢ given tag yᵢ? Viterbi decoding finds optimal tag sequence.

Learns from labeled data by counting transition and emission frequencies. First practical POS tagger. Accuracy ~96% on Penn Treebank.

Covered in depth in Section 6.
Discriminative Sequence
Conditional Random Field (CRF)

Discriminative model: directly models P(Y|X) without generating X. Can use arbitrary overlapping features (word shape, suffix, prefix, surrounding words). Avoids the label-bias problem of MEMMs. Uses Viterbi for decoding.

BiLSTM-CRF was the gold standard for NER from 2016–2018. The CRF layer enforces valid BIO sequences (prevents I-PER appearing after B-ORG).

Loss = negative log-likelihood of correct tag sequence, computed via forward algorithm.
Deep Learning
BiLSTM + CRF

Bidirectional LSTM generates context-rich token representations from both left and right context. CRF layer on top models tag dependencies. Character CNN captures morphological features (capitalization, suffix -ing, prefix un-).

End-to-end trainable. No feature engineering. ~91% NER F1 on CoNLL-2003 benchmark. Previously state-of-the-art; now superseded by BERT.

Architecture: Char CNN → word embedding → BiLSTM → CRF → BIO tags per token.
Transformer-Based
BERT Fine-tuning for Tagging

Fine-tune BERT with a token classification head: each subword token's output embedding → Linear(hidden_size, num_tags) → Softmax → predicted tag. For multi-subword words, use the FIRST subword's prediction.

BERT achieves ~93% NER F1 on CoNLL-2003 — surpassing BiLSTM-CRF by ~2%. Domain-specific BERT (BioBERT, LegalBERT) further improves on domain-specific NER.

Optional CRF on top of BERT further improves by enforcing valid BIO sequences.
Section 5

BIO Tagging Scheme and Sequence Tasks

5.1 The BIO Tagging Scheme

The BIO (Beginning-Inside-Outside) tagging scheme encodes entity or chunk spans as token-level labels. Every token in the sentence gets exactly one BIO label.

BIO / BIOES Tagging Schemes
BIO:
B-TYPE = Beginning of entity/chunk of TYPE ← first token of span
I-TYPE = Inside/continuation of TYPE ← subsequent tokens of span
O = Outside (not part of any span) ← non-entity token

BIOES (extended):
B-TYPE = Beginning of multi-token span
I-TYPE = Inside of multi-token span
O = Outside
E-TYPE = End of multi-token span
S-TYPE = Single-token span (alone)

Constraint: I-TYPE can only follow B-TYPE or I-TYPE of SAME type.
Invalid: I-PER after B-ORG (enforced by CRF layer)

Hidden Markov Model (HMM) for Sequence Tagging

6.1 HMM Architecture

The Hidden Markov Model (HMM) models sequence tagging as a generative process: the hidden states (POS tags) generate the observed tokens (words). We want to infer the most likely hidden state sequence given the observations.

  HMM for POS Tagging:

  HIDDEN STATES (tags):  y₁      y₂      y₃      y₄      y₅
                          DT  →   JJ  →   NN  →   VBZ →   IN
                          ↓       ↓       ↓       ↓       ↓
  OBSERVATIONS (words):  The   quick   cat    jumps   over
                         x₁      x₂     x₃      x₄      x₅

  Two probability tables:
  ┌──────────────────────────────────────────────────────────────┐
  │ TRANSITION MATRIX A: P(yᵢ | yᵢ₋₁)                         │
  │                                                              │
  │        DT    JJ    NN    VBZ   IN    ...                    │
  │ DT  [  0.05  0.30  0.55  0.02  0.02  ]  ← after DT: NN, JJ│
  │ JJ  [  0.00  0.10  0.75  0.05  0.00  ]  ← after JJ: NN    │
  │ NN  [  0.01  0.00  0.10  0.35  0.15  ]  ← after NN: VBZ   │
  │ VBZ [  0.20  0.05  0.20  0.00  0.20  ]  ← after VBZ: DT,IN│
  │ ...                                                          │
  └──────────────────────────────────────────────────────────────┘
  ┌──────────────────────────────────────────────────────────────┐
  │ EMISSION MATRIX B: P(xᵢ | yᵢ)                              │
  │                                                              │
  │        "The"  "cat"  "quick" "jumps" "over" ...            │
  │ DT   [  0.40   0.00   0.00    0.00    0.00  ]               │
  │ JJ   [  0.00   0.00   0.08    0.00    0.00  ]               │
  │ NN   [  0.00   0.12   0.01    0.01    0.00  ]               │
  │ VBZ  [  0.00   0.00   0.00    0.05    0.00  ]               │
  │ IN   [  0.00   0.00   0.00    0.00    0.07  ]               │
  └──────────────────────────────────────────────────────────────┘
Figure 6.1 — HMM for POS Tagging. Hidden states = POS tags, Observations = words. A = transition matrix, B = emission matrix.

6.2 HMM Training — Parameter Estimation

HMM Parameters — MLE from Labeled Corpus
Transition probability:
P(yᵢ = t₂ | yᵢ₋₁ = t₁) = C(t₁, t₂) / C(t₁)
C(t₁, t₂) = count of tag bigram (t₁→t₂) in training corpus
C(t₁) = count of tag t₁ in training corpus

Emission probability:
P(xᵢ = w | yᵢ = t) = C(w, t) / C(t)
C(w, t) = count of word w tagged as t in training corpus
C(t) = count of tag t in training corpus

Initial probability:
P(y₁ = t) = C(t at sentence start) / total sentences

Add Laplace smoothing to avoid zero probabilities:
P(w|t) = (C(w,t) + 1) / (C(t) + |V|)

6.3 Viterbi Algorithm — Decoding

The Viterbi algorithm is a dynamic programming algorithm that finds the most likely tag sequence Y* = argmax_Y P(Y|X). It computes the best path through the HMM trellis efficiently in O(|T|² × n) instead of the brute-force O(|T|ⁿ).

Viterbi Algorithm
vᵢ(t) = max_{t' ∈ T} [vᵢ₋₁(t') · P(t|t') · P(xᵢ|t)]

vᵢ(t) = probability of the most likely tag sequence ending in tag t at position i
P(t|t') = transition from previous tag t' to current tag t
P(xᵢ|t) = emission of word xᵢ from tag t

Backpointer: bᵢ(t) = argmax_{t'} [vᵢ₋₁(t') · P(t|t')]

Initialization: v₁(t) = P(y₁=t) · P(x₁|t) for each tag t
Forward pass: fill viterbi table left-to-right, O(|T|² × n)
Backward pass: trace back pointers from final state to recover Y*
Viterbi — Small Example: "The cat sat" (simplified)
Tags
T = {DT, NN, VBD} (simplified tagset)
v₁(DT) for "The"
P(DT|START) × P("The"|DT) = 0.40 × 0.40 = 0.160
v₁(NN) for "The"
P(NN|START) × P("The"|NN) = 0.30 × 0.00 = 0.000
v₂(NN) for "cat"
max(v₁(DT)×P(NN|DT), v₁(NN)×P(NN|NN)) × P("cat"|NN)
= max(0.160×0.60, 0) × 0.12 = 0.096 × 0.12 = 0.01152
v₃(VBD) for "sat"
max(v₂(NN)×P(VBD|NN)) × P("sat"|VBD) = best path
Best path
DT → NN → VBD (decoded via backpointers)
Tag sequence
"The"/DT "cat"/NN "sat"/VBD ✓

Syntax Trees — Constituency Parsing

7.1 What is a Syntax Tree?

A syntax tree (parse tree) is a hierarchical representation of a sentence's grammatical structure. It shows how words group into phrases, and how phrases combine to form the complete sentence. Syntax trees reveal the grammatical relationships and compositional meaning structure of language.

7.2 Constituency Parse Tree

A constituency parse tree organizes the sentence into nested phrase constituents. Each internal node is a phrase type (NP, VP, PP), each leaf is a POS-tagged word.

  Sentence: "The quick fox sat on the mat"

                          S
                    ┌─────┴──────────┐
                   NP               VP
              ┌────┼────┐       ┌────┴──────┐
             DT   JJ   NN     VBD          PP
             │    │    │       │       ┌────┴────┐
            The quick fox    sat      IN        NP
                                      │      ┌───┴───┐
                                     on     DT      NN
                                            │       │
                                           the     mat

  Phrase types:
  S   = Sentence       (the whole sentence)
  NP  = Noun Phrase    (the quick fox, the mat)
  VP  = Verb Phrase    (sat on the mat)
  PP  = Prepositional Phrase (on the mat)
  DT  = Determiner    (the, a)
  JJ  = Adjective     (quick)
  NN  = Noun          (fox, mat)
  VBD = Past tense VB (sat)
  IN  = Preposition   (on)
Figure 7.1 — Constituency parse tree. Hierarchical phrase structure. Every leaf is a word; every internal node is a phrase or clause category.

Exam Point — Structural Ambiguity in Parse Trees

Sentences can be syntactically ambiguous — having two valid parse trees with different meanings. "I saw the man with the telescope" has two parses: (1) [VP saw [NP the man [PP with the telescope]]] — the man has the telescope; (2) [VP saw [NP the man] [PP with the telescope]] — I used the telescope to see. Resolving such ambiguity requires semantic and world knowledge beyond syntax.

Dependency Parsing

8.1 Dependency Grammar

In dependency grammar, every word (except the root) depends on exactly one other word — called its head. The labeled directed arc from head to dependent encodes the grammatical relation between them (subject, object, modifier, etc.).

  Sentence: "The quick fox sat on the mat"

    ┌─────────────────nsubj──────────────────┐
    │         ┌─det──┐     ┌─amod──┐         │
    │         │      │     │       │         │
   sat ←─────fox ←─The   fox ←──quick      sat
    │                                        │
    └────prep─────── on ──── pobj ───── mat ─┘
                               │
                            ┌──┘
                           mat ←─det── the

  Universal Dependency Relations:
  nsubj  = nominal subject (fox is subject of sat)
  det    = determiner (The modifies fox; the modifies mat)
  amod   = adjectival modifier (quick modifies fox)
  prep   = prepositional modifier (on modifies sat)
  pobj   = prepositional object (mat is object of on)

  ROOT: sat (the main predicate)
  Every other word has exactly one head.
Figure 8.1 — Dependency parse. Arrows from head → dependent. Labels show the grammatical relation. Root = main verb.

8.2 Universal Dependency Relations

  • ▸
    nsubj (nominal subject): The nominal subject of a verb. "The cat chased the dog" → cat is nsubj of chased.
  • ▸
    dobj (direct object): The direct object of a verb. "She ate the apple" → apple is dobj of ate.
  • ▸
    amod (adjectival modifier): An adjective modifying a noun. "quick fox" → quick is amod of fox.
  • ▸
    advmod (adverbial modifier): An adverb modifying a verb or adjective. "runs quickly" → quickly is advmod of runs.
  • ▸
    prep (prepositional modifier): A preposition modifying a verb/noun. "sat on the mat" → on is prep of sat.
  • ▸
    det (determiner): A determiner modifying a noun. "the cat" → the is det of cat.
  • ▸
    conj (conjunct): Two words connected by a conjunction. "cats and dogs" → dogs is conj of cats.
  • ▸
    cc (coordinating conjunction): The conjunction itself. "cats and dogs" → and is cc of cats.

Context-Free Grammars (CFG)

9.1 Formal Definition of CFG

A Context-Free Grammar (CFG) is a formal grammar system that captures the hierarchical structure of natural language (and programming languages). It defines a set of production rules specifying how symbols can be rewritten.

CFG Formal Definition

A CFG is a 4-tuple G = (N, Σ, R, S) where:

N = set of non-terminal symbols (phrase types: S, NP, VP, PP, N, V...)

Σ = set of terminal symbols (actual words: the, cat, sat, on...)

R = set of production rules of form A → α where A ∈ N and α ∈ (N ∪ Σ)*

S = start symbol (always S, representing a complete sentence)

"Context-free" means: the left-hand side A can be rewritten to α regardless of the surrounding context — only A itself matters, not what surrounds it.

9.2 Example CFG for English

A Simple English CFG — Production Rules
Phrasal Rules (non-terminal → sequence of non-terminals):
S → NP VP ← sentence = noun phrase + verb phrase
S → NP VP PP ← sentence with prepositional modifier
NP → DT N ← noun phrase = determiner + noun
NP → DT JJ N ← noun phrase with adjective
NP → NP PP ← noun phrase with prepositional phrase
VP → V ← intransitive verb
VP → V NP ← transitive verb + object
VP → V NP PP ← verb + object + prep phrase
VP → VP PP ← VP with prepositional modifier
PP → P NP ← prep phrase = preposition + noun phrase

Lexical Rules (non-terminal → terminal word):
DT → "the" | "a" | "an" | "this" | "that"
N → "cat" | "dog" | "mat" | "fox" | "table"
JJ → "quick" | "big" | "lazy" | "brown"
V → "sat" | "chased" | "put" | "saw"
P → "on" | "in" | "over" | "under" | "with"

9.3 Deriving a Sentence from CFG

CFG Derivation — "The quick fox sat on the mat"
Step 1
S → NP VP (apply S → NP VP)
Step 2
NP VP → DT JJ N VP (apply NP → DT JJ N)
Step 3
DT JJ N VP → "The" JJ N VP (apply DT → "The")
Step 4
"The" JJ N VP → "The" "quick" N VP (apply JJ → "quick")
Step 5
"The" "quick" N VP → "The" "quick" "fox" VP (apply N → "fox")
Step 6
...VP → VP PP → V PP → "sat" PP (apply VP → V, V → "sat")
Step 7
PP → P NP → "on" NP → "on" DT N → "on" "the" "mat"
Result
"The quick fox sat on the mat" ✓ (valid derivation from S)

9.4 Probabilistic CFG (PCFG)

A CFG with a single sentence can often have multiple parse trees (structural ambiguity). A Probabilistic CFG (PCFG) assigns a probability to each rule. The probability of a parse tree is the product of all rule probabilities used, and the most probable parse tree is preferred.

Probabilistic Context-Free Grammar (PCFG)
Each rule A → α has a probability P(A → α)
Constraint: For each A, Σ_{α} P(A → α) = 1 ← probabilities sum to 1

Example probabilities:
P(NP → DT N) = 0.50
P(NP → DT JJ N) = 0.30
P(NP → NP PP) = 0.20
(must sum to 1.00 for NP rules)

Parse tree probability:
P(tree) = Π_{rules used} P(A → α)

Best parse: Y* = argmax_{trees} P(tree)
Computed efficiently by CYK algorithm (O(n³|G|))

9.5 CFG Limitations for Natural Language

  • ▸
    Independence assumption: CFG rules expand non-terminals independently of context. But "NP → DT N" fires the same way whether the NP is a subject or object — but subject NPs prefer pronouns while objects prefer full NPs. Context matters.
  • ▸
    Lexical dependency blindness: "Ate spaghetti with a fork" (manner) vs "Ate spaghetti with meatballs" (ingredient) — the PP attaches differently based on the object, not the VP structure. CFG doesn't capture this lexical preference.
  • ▸
    Combinator explosion: Accurate natural language grammars have thousands of rules. Parsing is O(n³|G|) — grows with grammar size.
  • ▸
    Solutions: Lexicalized PCFGs (attach head word to each non-terminal). Neural parsers (directly learn to parse from trees without explicit grammar rules).
Section 10

CFG Parsing Algorithms

10.1 CYK Algorithm (Cocke-Younger-Kasami)

The CYK algorithm is a bottom-up dynamic programming parser for CFGs in Chomsky Normal Form (CNF). It fills a triangular chart (parse table) to determine all possible parse trees for a sentence.

Chomsky Normal Form (CNF) — Required for CYK

Every production must have one of two forms: A → BC (two non-terminals) or A → a (one terminal). Any CFG can be converted to CNF without changing the language it generates. CNF conversion: eliminate unit rules (A→B), eliminate ε-rules, binarize rules with 3+ symbols on RHS.

CYK Algorithm — Bottom-Up Chart Parsing
Input: sentence w₁w₂...wₙ, Grammar G in CNF
Table: table[i][j] = set of non-terminals that can span wᵢ...wⱼ

Initialization (span of 1):
for each word wᵢ:
table[i][i] = {A : A → wᵢ is a rule in G}

Fill table (span length 2 to n):
for length = 2 to n:
for start = 1 to n-length+1:
end = start + length - 1
for split = start to end-1:
for each rule A → BC in G:
if B ∈ table[start][split] AND C ∈ table[split+1][end]:
table[start][end] += {A}

Accept if S ∈ table[1][n] ← S spans the whole sentence
Complexity: O(n³ |G|) ← cubic in sentence length
CYK Table — "The cat sat" (simplified CNF grammar)
Rules
S→NP VP | NP→DT NN | VP→V | DT→"The" | NN→"cat" | V→"sat"
table[1][1]
{DT} ← "The" → DT
table[2][2]
{NN} ← "cat" → NN
table[3][3]
{V} ← "sat" → V
table[1][2]
{NP} ← NP → DT NN (DT∈[1,1], NN∈[2,2]) ✓
table[2][3]
{VP} ← VP → V alone? Need CNF — VP → V only if V is a terminal rule
table[1][3]
{S} ← S → NP VP (NP∈[1,2], VP∈[3,3]) ✓
Result
S ∈ table[1][3] → sentence is grammatical ✓

10.2 Earley Algorithm

  • ▸
    Advantage over CYK: Works on any CFG (not just CNF). Top-down with bottom-up dynamic programming. O(n³) in general, O(n²) for unambiguous grammars, O(n) for LR(1) grammars.
  • ▸
    Earley items: Dotted rules [A → α • β, i] meaning "we've matched α starting at position i, and are looking for β next." Items advance as each word is processed.
  • ▸
    Three operations: Scanner (match a word against the next expected token), Predictor (add items for rules that could apply next), Completer (when a constituent is complete, advance all waiting items).

Section 11

Chunking — Shallow Parsing

11.1 What is Chunking?

Chunking (also called shallow parsing or partial parsing) identifies flat, non-recursive phrase-level constituents in a sentence without producing a full parse tree. It groups adjacent POS-tagged tokens into non-overlapping chunks based on regex-like patterns over POS tags.

Chunking vs Full Parsing

Full Parsing (deep): Produces a complete hierarchical tree (recursive NPs inside PPs inside VPs). Expensive, error-prone for complex sentences. Required for full semantic interpretation.

Chunking (shallow): Produces flat, non-recursive noun phrases, verb phrases, and prepositional phrases. Fast, robust, sufficient for many information extraction tasks. Does NOT capture recursive embedding (no NP inside PP inside NP).

11.2 Noun Phrase (NP) Chunking

The most common chunking task. NP chunks are simple noun phrases — typically DT? JJ* NN+ (optional determiner, zero or more adjectives, one or more nouns).

Sentence: "The quick brown fox jumped over a lazy dog"

The quick brown foxNP jumped over a lazy dogNP
Python — NLTK RegexpParser for Chunking
import nltk from nltk import RegexpParser, pos_tag, word_tokenize sentence = "The quick brown fox jumped over a lazy dog near the river" tokens = word_tokenize(sentence) pos_tags = pos_tag(tokens) print(pos_tags) # → [('The','DT'),('quick','JJ'),('brown','JJ'),('fox','NN'),('jumped','VBD'),...] # ── Define chunk grammar using regex over POS tags ────────────── grammar = """ NP: {<DT>?<JJ>*<NN>+} # optional DT, 0+ JJ, 1+ NN {<NNP>+} # proper nouns together PP: {<IN><NP>} # preposition + NP VP: {<VB.*><NP|PP|CLAUSE>+} # verb followed by NP or PP """ chunker = RegexpParser(grammar) tree = chunker.parse(pos_tags) # Display the parse tree print(tree) tree.draw() # opens graphical tree view # Extract NP chunks as text for subtree in tree.subtrees(filter=lambda t: t.label() == 'NP'): print("NP:", ' '.join(word for word, tag in subtree.leaves())) # → NP: The quick brown fox # → NP: a lazy dog # → NP: the river

from nltk.chunk import tree2conlltags grammar = """ NP: {<DT>?<JJ>*<NN>+} VP: {<VB.*>} PP: {<IN><NP>} """ chunker = RegexpParser(grammar) tree = chunker.parse(pos_tags) # Convert tree back to IOB tags iob_tags = tree2conlltags(tree) for word, pos, chunk in iob_tags: print(f"{word:12} {pos:6} {chunk}") # The DT B-NP # quick JJ I-NP # brown JJ I-NP # fox NN I-NP # jumped VBD B-VP # over IN B-PP # a DT B-NP # lazy JJ I-NP # dog NN I-NP # Evaluate chunker on CoNLL-2000 dataset from nltk.corpus import conll2000 print(chunker.accuracy(conll2000.chunked_sents('test.txt')))
Section 12

Chinking — Excluding Tokens from Chunks

12.1 What is Chinking?

Chinking is the complement of chunking — it defines what to exclude from an existing chunk, rather than what to include. A chink is a sequence of tokens that is split or removed from a chunk. Chinking is used to refine chunks by removing specific word types that should not be part of the chunk.

Chunking vs Chinking — The Key Distinction

Chunking (positive rule with {}): Specifies what tokens to GROUP INTO a chunk. "Find all DT JJ* NN sequences and call them NP chunks."

Chinking (negative rule with }{): Specifies what tokens to SPLIT OUT of an existing chunk. "Within the chunks found above, remove any VBD or IN tokens, splitting the chunk at those points."

Chinking is useful when it's easier to define what should NOT be in a chunk than what should be. You first create a large chunk (perhaps the whole sentence) and then remove specific token types.

12.2 Chinking — How it Works

Chinking Syntax in NLTK RegexpParser
Chunking rule (include): NP: {<DT>?<JJ>*<NN>+}
Chinking rule (exclude): NP: }<VBD|IN>{

A chinking rule uses REVERSED BRACES: }<pattern>{
This removes tokens matching the pattern from any existing chunk.

Combined example:
CLAUSE: {<.*>+} ← first, chunk EVERYTHING into one big clause
}<VBD|IN>{ ← then REMOVE all VBD and IN tokens from chunks

Effect on "The big fox sat on the mat":
Before: {The big fox sat on the mat} ← one chunk
After: {The big fox} sat {on} {the mat}
← "sat"(VBD) splits the chunk; "on"(IN) splits a sub-chunk

12.3 Chinking Worked Example

Chinking — Step by Step: "The cat quickly sat on a mat"
POS tags
The/DT cat/NN quickly/RB sat/VBD on/IN a/DT mat/NN
Step 1: Chunk ALL
{The/DT cat/NN quickly/RB sat/VBD on/IN a/DT mat/NN} ← one big chunk
Chink rule
}<VBD|RB|IN>{ ← remove VBD, RB, and IN tokens from chunks
Remove "sat" (VBD)
{The/DT cat/NN quickly/RB} sat/VBD {on/IN a/DT mat/NN}
Remove "quickly" (RB)
{The/DT cat/NN} quickly/RB sat/VBD {on/IN a/DT mat/NN}
Remove "on" (IN)
{The/DT cat/NN} quickly/RB sat/VBD on/IN {a/DT mat/NN}
Final chunks
[NP: The cat] quickly sat on [NP: a mat]
Python — Chinking with NLTK RegexpParser
import nltk from nltk import RegexpParser, pos_tag, word_tokenize sentence = "The cat quickly sat on a comfortable mat near the river" tokens = word_tokenize(sentence) pos_tags = pos_tag(tokens) # Grammar with CHUNKING + CHINKING combined grammar = """ NP: {<.*>+} # Step 1: chunk everything }<VBD|VBZ|VBN|VBG|VBP|RB|IN|CC|DT>{ # Step 2: chink (remove) verbs, adverbs, preps, conj, det """ chunker = RegexpParser(grammar) tree = chunker.parse(pos_tags) print(tree) # (S # (NP The/DT cat/NN) # quickly/RB # sat/VBD # on/IN # (NP comfortable/JJ mat/NN) ← "a" chinking removed DT # near/IN # (NP river/NN)) ← "the" chinking removed DT # Compare: pure chunking approach grammar2 = """ NP: {<DT>?<JJ>*<NN>+} # pure chunking: include DT JJ* NN+ """ chunker2 = RegexpParser(grammar2) tree2 = chunker2.parse(pos_tags) print(tree2) # Different result: includes DT inside NP

12.4 Chunking and Chinking — Applications

  • ▸
    Information Extraction: NP chunks = candidate entities. VP chunks = candidate relations/events. PP chunks = location/time/manner information. Chunked text can be searched for patterns without full parsing.
  • ▸
    Keyword/Keyphrase Extraction: NP chunks are the best candidates for topic keywords. "machine learning algorithms" is a much better keyword extracted as one NP chunk than individually as "machine", "learning", "algorithms".
  • ▸
    Named Entity Extraction (rule-based): Chunked NPs that contain capitalized words and are followed by certain contexts are often named entities. Chunking + patterns is faster than full NER for restricted domains.
  • ▸
    Text Summarization (extractive): Important NP chunks (high TF-IDF NPs) can be extracted as keyphrases for document summary. Avoids extracting grammatically fragmented key terms.
Part 6 — Automatic Paraphrasing of Texts
Section 13

Automatic Paraphrasing of Texts

13.1 What is Automatic Paraphrasing?

Automatic paraphrasing is the task of generating a new text that expresses the same meaning as the original using different words, phrases, or syntactic structures. The paraphrase must preserve the semantic content while varying the surface form — changing vocabulary, word order, or grammatical structure.

Paraphrasing — The Semantic Equivalence Requirement

A valid paraphrase preserves meaning: "The dog chased the cat" ↔ "The cat was chased by the dog" (active/passive transformation — same meaning, different syntax). But "The cat chased the dog" is NOT a paraphrase despite identical words (agent and patient swapped — different meaning). Paraphrasing operates at the semantic level, not just lexical substitution.

13.2 Types of Paraphrases

Type 1
Lexical Paraphrases

Word substitution: Replace words with synonyms or semantically equivalent phrases. "The automobile broke down" → "The car stopped working." "She is intelligent" → "She is smart/clever/bright."

Sources: WordNet synonym sets, word embedding similarity (word2vec, GloVe), contextual embeddings (BERT masked prediction for synonym suggestions).

Challenge: Synonyms are context-dependent. "Large" and "big" are synonyms in most contexts but not "large-scale" vs "big-scale".
Type 2
Syntactic Paraphrases

Structural transformation: Change grammatical structure while preserving meaning. Active ↔ Passive: "The police arrested the suspect" ↔ "The suspect was arrested by the police." Nominalization: "He decided to leave" ↔ "His decision was to leave."

Methods: CFG-based transformation rules applied to parse trees. TreeSubstitutionGrammar. Synchronous CFGs that map source to target syntactic structures.

Requires full parse tree. Transformations must be semantics-preserving (not all syntactic changes preserve meaning).
Type 3
Phrasal Paraphrases

Multi-word to multi-word: Replace idioms, collocations, or multi-word expressions with equivalents. "kick the bucket" → "pass away." "blow one's top" → "become angry." "in spite of" → "despite."

Extraction: Paraphrase databases (PPDB — Paraphrase Database, 100M+ paraphrase pairs extracted via bilingual pivoting). MultiParaphrase Corpus. Wikipedia sentence alignment.

PPDB provides scored paraphrase pairs at word, phrase, and sentence level.
Type 4
Discourse Paraphrases

Sentence restructuring: Combine or split sentences while preserving overall meaning. "She was tired. She went to bed early." ↔ "Because she was tired, she went to bed early." "Despite being exhausted, she worked late" ↔ "She was exhausted. Nevertheless, she worked late."

Requires understanding of discourse structure, coherence relations, and sentence boundaries.

Most complex form of paraphrasing. Requires coreference resolution and discourse parsing.

13.3 Bilingual Pivoting for Paraphrase Extraction

The most powerful automatic method for extracting paraphrase pairs uses multilingual resources: if two English phrases translate to the same phrase in another language (the "pivot"), they are likely paraphrases of each other.

  BILINGUAL PIVOTING (using French as pivot language):

  English Phrase A         French (pivot)         English Phrase B
  "automobile"     →  →   "voiture"   ←  ←   "car"
  "begin"          →  →   "commencer" ←  ←   "start"
  "kick the bucket"→  →   "décéder"   ←  ←   "pass away"

  Logic: If English_A and English_B both translate to the same French_pivot,
         then English_A ≈ English_B (they express the same meaning)

  Sources: Parallel corpora (EuroParl, UN corpus), translation memories
  PPDB (Paraphrase Database): extracted this way from 23 language pairs
  Result: "automobile" and "car" are identified as paraphrases
          "in the vicinity of" and "near" are identified as paraphrases
Figure 13.1 — Bilingual pivoting. Two source phrases that map to the same translation are candidate paraphrases.

13.4 Neural Paraphrasing Methods

  • ▸
    Seq2Seq paraphrasing: Train an encoder-decoder Transformer on pairs of paraphrases (from PPDB, Quora Question Pairs, Twitter URL pairs). At inference: encode source → decode to paraphrase. Can generate diverse paraphrases by beam search or sampling.
  • ▸
    Round-trip translation: Translate English → French → back to English. The re-translation is often a natural paraphrase of the original. Multiple pivot languages produce diverse paraphrases. Used extensively in NLP data augmentation.
  • ▸
    PEGASUS / T5 for paraphrasing: Fine-tune sequence-to-sequence models on paraphrase pairs. The model learns to generate meaning-equivalent reformulations. Controllable paraphrasing: prefix with "Formal:", "Casual:", "Technical:" to guide the style.
  • ▸
    GPT-based paraphrasing: Prompt: "Paraphrase the following sentence: [sentence]". In-context examples guide style. LLMs like GPT-4 produce high-quality, diverse paraphrases across levels (lexical, syntactic, discourse) in a single model.

13.5 Applications of Automatic Paraphrasing

  • ▸
    Data augmentation: Generate additional training examples for low-resource NLP tasks. A classifier trained on "Is this spam?" also learns from "Is this unsolicited email?" if augmented with paraphrases. Particularly valuable for question answering, NLI, and text classification.
  • ▸
    Plagiarism detection: Detect paraphrased plagiarism (where students rewrite source text). Match not just word overlap but semantic similarity using paraphrase-aware models.
  • ▸
    Question generation: Generate multiple rephrasings of the same question for diverse FAQ systems, quiz generation, and dialogue variation.
  • ▸
    Readability simplification: Generate simpler paraphrases of complex sentences (reduce reading level, explain technical jargon). "The patient exhibited febrile symptoms" → "The patient had a fever."
  • ▸
    Machine translation evaluation: METEOR score uses paraphrase matching to evaluate MT output more flexibly than BLEU's exact n-gram matching. Paraphrase pairs allow partial credit for semantically equivalent translations.
  • ▸
    Adversarial testing: Generate paraphrases of test inputs to check model robustness. A robust NLU model should give the same prediction for all paraphrases of a sentence.

13.6 Paraphrase Evaluation

  • ▸
    Automatic metrics: BLEU (n-gram overlap with reference paraphrases), METEOR (synonym-aware n-gram), BERTScore (contextual embedding similarity), ROUGE (recall-oriented n-gram overlap). All imperfect — high BLEU doesn't guarantee semantic equivalence.
  • ▸
    Semantic similarity: Cosine similarity of sentence embeddings (SBERT, InferSent). A good paraphrase should have high similarity (>0.85) to the original in embedding space.
  • ▸
    Human evaluation: Adequacy (does it mean the same?) + Fluency (is it grammatical and natural?) + Diversity (how different is the wording?). The gold standard — automatic metrics remain approximations.

Module 4 — Complete Summary

  • 1.
    Language Modeling: Assigns P(sentence). N-gram LM: P(wₙ|context) ≈ P(wₙ|wₙ₋ₙ₊₁...wₙ₋₁) = C(context,wₙ)/C(context). Perplexity = P(W)^{−1/N}: lower = better. Smoothing handles zero counts (Add-1, Kneser-Ney). Neural LMs (RNN, LSTM, Transformer) outperform N-gram LMs significantly.
  • 2.
    Sequence Tagging: Assigns one label per token (not one per document). Tasks: POS tagging, NER, chunking, slot filling. Challenge: tags depend on adjacent tags — must predict the best joint tag sequence Y* = argmax P(Y|X), not just token-by-token argmax.
  • 3.
    BIO Scheme: B-TYPE = beginning of entity span, I-TYPE = inside span, O = not an entity. BIOES extends with E (end) and S (single-token). Span extraction: B starts span, I continues, O or new B ends span. Enforced as constraint by CRF layer.
  • 4.
    HMM for Tagging: Two tables: A (transitions P(yᵢ|yᵢ₋₁)) and B (emissions P(xᵢ|yᵢ)). Trained by counting tag bigrams and word-tag pairs from labeled corpus (with Laplace smoothing). Viterbi: O(|T|²×n) DP finds optimal tag sequence via backtracking.
  • 5.
    Syntax Trees: Constituency tree: hierarchical phrase structure (S→NP VP, NP→DT JJ NN). Dependency tree: binary head→modifier relations (nsubj, dobj, amod, prep). Ambiguity: one sentence can have multiple parse trees with different meanings.
  • 6.
    CFG: G=(N,Σ,R,S). Production rules: A→BC (binary branching) or A→a (lexical). CYK algorithm: bottom-up DP in O(n³|G|) on CNF grammars. PCFG: each rule has probability; P(tree)=Π P(rules). Most probable parse resolves structural ambiguity.
  • 7.
    Chunking: Shallow/partial parsing — identifies flat, non-recursive NP, VP, PP constituents using regex patterns over POS tags. NLTK RegexpParser: {<DT>?<JJ>*<NN>+} includes token sequences; BIO tags encode chunk boundaries. Faster and more robust than full parsing for IE tasks.
  • 8.
    Chinking: Complement of chunking — specifies what to EXCLUDE from chunks using reversed braces }<pattern>{. Process: (1) chunk everything into one large chunk, (2) chink removes specified POS types (verbs, prepositions, conjunctions), splitting the chunk at those points. Produces the same result as chunking but defined negatively.
  • 9.
    Automatic Paraphrasing: Generate semantically equivalent text with different surface form. Types: Lexical (synonym substitution), Syntactic (active↔passive, nominalization), Phrasal (multi-word substitution), Discourse (sentence restructuring). Methods: bilingual pivoting (PPDB), round-trip translation, seq2seq models, LLM prompting. Applications: data augmentation, plagiarism detection, readability simplification.
