Text Mining — What, Why and How

1.1 What is Text Mining?

Text Mining (also called Text Analytics or Knowledge Discovery from Text — KDT) is the process of deriving high-quality, structured information and patterns from large volumes of unstructured or semi-structured text data through a combination of NLP, statistical analysis, and machine learning techniques.

Text Mining vs Traditional Data Mining

Traditional data mining operates on structured, tabular data (rows and columns with defined types). Text mining operates on unstructured raw text where structure must first be discovered or imposed. The core challenge: convert free-flowing human language into a form that algorithms can process and extract patterns from.

1.2 Text Mining Tasks

  • ▸
    Information Extraction: Extract structured facts — entities (people, organizations, places), relationships between them, events. Turn "Apple Inc. acquired Beats in 2014 for $3B" into a structured triple: (Apple Inc., acquired, Beats, 2014, $3B).
  • ▸
    Text Classification: Assign predefined labels to documents — spam/not-spam, sentiment polarity, topic category, language identification. Requires a labeled training corpus and a learned classifier.
  • ▸
    Text Clustering: Group similar documents without predefined labels — unsupervised discovery of topics or themes. K-means on TF-IDF vectors, Latent Dirichlet Allocation (LDA) for topic modeling.
  • ▸
    Summarization: Produce a concise summary of one or many documents, retaining the most important information. Extractive (select key sentences) or abstractive (generate new text).
  • ▸
    Question Answering: Given a corpus and a natural language question, locate or generate the answer from the corpus. Requires understanding both the question and documents.
  • ▸
    Opinion Mining (Sentiment Analysis): Detect opinions, sentiments, and subjectivity in text. Particularly valuable for product reviews, social media monitoring, and brand analytics.

Section 2

Text Cleaning Pipeline

2.1 Why Cleaning is Critical

Raw text from real-world sources is messy — it contains HTML tags, special characters, inconsistent encoding, URLs, numbers, extra whitespace, typos, and noise specific to the source (emojis in tweets, code in Stack Overflow posts, headers in legal documents). Models trained on noisy text learn noise as signal.

2.2 Text Cleaning Steps — In Order

Worked Example — Cleaning a Raw Tweet
Raw input
"<p>I LOVE this product!! 😍😍 Visit https://t.co/xyz #BestEver @john123 it's AMAZING!!!! 100%"
1. HTML strip
"I LOVE this product!! 😍😍 Visit https://t.co/xyz #BestEver @john123 it's AMAZING!!!! 100%"
2. URL removal
"I LOVE this product!! 😍😍 #BestEver @john123 it's AMAZING!!!! 100%"
3. Mention/hashtag
"I LOVE this product!! 😍😍 it's AMAZING!!!! 100%"
4. Emoji handling
"I LOVE this product!! [heart_eyes] [heart_eyes] it's AMAZING!!!! 100%"
5. Lowercasing
"i love this product!! [heart_eyes] [heart_eyes] it's amazing!!!! 100%"
6. Punctuation clean
"i love this product heart_eyes heart_eyes its amazing 100"
7. Whitespace normalize
"i love this product heart_eyes heart_eyes its amazing 100"
8. Contraction expand
"i love this product heart_eyes heart_eyes it is amazing 100"

2.3 Cleaning Operations — Detailed

  • ▸
    HTML/XML tag removal: Use regex or BeautifulSoup. <b>Important</b> → "Important". Web-scraped text is full of tags — must be stripped before any linguistic processing.
  • ▸
    URL and email removal: URLs carry no semantic content for most NLP tasks (unless the task is about URLs). Replace with a placeholder [URL] or remove entirely. Regex: https?://\S+.
  • ▸
    Encoding normalization: Convert to UTF-8. Handle smart quotes (" " → "), en/em dashes, accented characters (café → cafe, or keep as-is). Unicode normalization (NFC/NFD) ensures consistent byte representations.
  • ▸
    Lowercasing: Convert all text to lowercase. "Apple" and "apple" should map to the same token in most tasks. Exception: For NER, case carries information — "Apple" (company) vs "apple" (fruit). For deep learning models, avoid lowercasing — the model handles it.
  • ▸
    Number handling: Depending on the task: remove numbers (sentiment, topic classification), replace with placeholder [NUM] (general NLP), normalize to word form (one, two, three), or keep (financial text, date extraction).
  • ▸
    Punctuation removal: For Bag-of-Words/TF-IDF: remove punctuation with string.punctuation. For sequence models (BERT, LSTM): keep punctuation — it carries syntactic and semantic information (question marks, commas, ellipses all matter).
  • ▸
    Contraction expansion: "can't" → "cannot", "it's" → "it is", "won't" → "will not". Important for consistent tokenization. Libraries: contractions (Python), pycontractions.
  • ▸
    Emoji handling: For social media tasks: convert to text description using emoji library (😍 → "heart_eyes") — emojis carry strong sentiment signals. For formal text: remove. Never ignore silently — they may dominate the signal.
  • ▸
    Extra whitespace: Multiple spaces, tabs, newlines collapse to a single space. Use re.sub(r'\s+', ' ', text).strip().
Task-Dependent Cleaning — No One-Size-Fits-All

Cleaning decisions must be task-specific. For a legal contract NER system: preserve case (entities are capitalized), keep numbers (dates, amounts), keep punctuation (clause boundaries). For a spam classifier: remove URLs and numbers. For a social media sentiment model: convert emojis to text, expand slang. Never apply the same cleaning pipeline to every task blindly.

2.4 Text Cleaning with Python — Code

Python — Text Cleaning Utilities
import re, unicodedata, string from contractions import fix def clean_text(text): # 1. Decode and normalize Unicode text = unicodedata.normalize('NFKD', text) # 2. Remove HTML tags text = re.sub(r'<[^>]+>', '', text) # 3. Remove URLs text = re.sub(r'https?://\S+|www\.\S+', '', text) # 4. Expand contractions text = fix(text) # 5. Lowercase text = text.lower() # 6. Remove punctuation (task-dependent) text = text.translate(str.maketrans('', '', string.punctuation)) # 7. Remove extra whitespace text = re.sub(r'\s+', ' ', text).strip() return text

Tokenization — Concepts and Importance

3.1 What is Tokenization?

Tokenization is the process of splitting raw text into smaller units called tokens, which serve as the fundamental units of processing in any NLP pipeline. A token can be a word, subword, character, sentence, or any other meaningful linguistic unit depending on the tokenization strategy.

Why Tokenization is Foundational

Every downstream NLP operation — frequency analysis, stemming, POS tagging, NER, embedding lookup, language model training — operates on tokens. The quality and granularity of tokenization directly determines what patterns the model can learn. A poor tokenization choice can prevent a model from learning even the simplest patterns correctly.

3.2 Levels of Tokenization

LevelToken UnitExampleUse CasePros / Cons
Sentence Complete sentence "I love NLP." → ["I love NLP.", "It's fascinating."] Summarization, sentence embeddings, discourse + Simple. − Sentence boundary detection is non-trivial (Dr. Smith vs end-of-sentence ".")
Word Whitespace/punctuation-separated word "I love NLP" → ["I", "love", "NLP"] Classical NLP, BoW, TF-IDF, Word2Vec + Intuitive. − Large vocabulary; OOV problem; fails on compound words, contractions
Subword Frequent subword units (BPE/WP/SP) "unhappiness" → ["un", "##happi", "##ness"] BERT, GPT, modern Transformers — universal standard + No OOV; handles morphology, multilingual. − Less intuitive; tokens ≠ words
Character Individual character "NLP" → ["N", "L", "P"] Spelling correction, morphology, low-resource languages + No OOV, handles typos. − Very long sequences; harder to learn semantic units
Byte UTF-8 byte All text → 256-token vocabulary ByT5, byte-level models + Universal, no tokenizer needed. − Longest sequences; limited semantic unit

3.3 Tokenization Challenges

  • ▸
    Contractions: "don't" — one word or two ("do", "n't" or "do", "not")? Different tokenizers handle this differently. The choice affects downstream tasks like POS tagging.
  • ▸
    Hyphenated words: "state-of-the-art" — one token or four? Domain-dependent: keep as one in technical text, split for general purposes.
  • ▸
    Abbreviations and periods: "Dr. Smith went to Washington, D.C." — periods are both sentence-ending and part of abbreviations. Must distinguish sentence boundaries from abbreviation dots.
  • ▸
    URLs and Emails: Should not be split on punctuation — "user@email.com" is one token, not five.
  • ▸
    Non-Western languages: Chinese, Japanese, Korean, Thai have no spaces between words — word boundary detection requires a segmentation model, not just whitespace splitting.
  • ▸
    Special characters: "@user", "#hashtag", "$100.50", ":)" — all need special handling depending on domain.
Section 4

Various Tokenizers — Types and Algorithms

4.1 Rule-Based / Whitespace Tokenizers

  • ▸
    Whitespace tokenizer: Split on any whitespace. Simplest possible. Fails on punctuation ("word." becomes ["word."] not ["word", "."]). Only suitable for very clean, pre-processed text.
  • ▸
    Punctuation + whitespace tokenizer: Split on whitespace and punctuation. Fast, deterministic. Handles most simple cases. Still fails on contractions, abbreviations, domain-specific terms.
  • ▸
    Penn Treebank Tokenizer (NLTK's word_tokenize): Rule-based tokenizer designed for English following Penn Treebank conventions. Splits contractions ("don't" → ["do", "n't"]), handles punctuation intelligently, respects some abbreviations.
  • ▸
    Regex tokenizer: Define a pattern for what constitutes a token using regular expressions. Highly flexible for specialized text (e.g., keep hashtags and @mentions intact for social media).
Python — Rule-Based Tokenizers (NLTK)
from nltk.tokenize import word_tokenize, sent_tokenize, RegexpTokenizer text = "Dr. Smith can't visit Washington D.C. today!" # NLTK word tokenizer (Penn Treebank style) word_tokenize(text) # → ['Dr.', 'Smith', 'ca', "n't", 'visit', 'Washington', 'D.C.', 'today', '!'] # Sentence tokenizer (Punkt algorithm) sent_tokenize("Hello Dr. Smith. How are you? I am fine.") # → ['Hello Dr. Smith.', 'How are you?', 'I am fine.'] # Regex tokenizer — keep only alphanumeric words tok = RegexpTokenizer(r'\w+') tok.tokenize(text) # → ['Dr', 'Smith', 'can', 't', 'visit', 'Washington', 'D', 'C', 'today']

4.2 Statistical / ML-Based Tokenizers

  • ▸
    Punkt Sentence Tokenizer (NLTK): Unsupervised algorithm that learns from text to distinguish sentence-ending periods from abbreviation periods. Trained on a corpus, then applied. Much better than simple period-splitting.
  • ▸
    SpaCy tokenizer: Rules + exception dictionary. Handles contractions, punctuation, URLs intelligently per language. Fast (C extension). Used for production English NLP pipelines.

4.3 Subword Tokenizers — The Modern Standard

Subword tokenizers address two key problems with word-level tokenization: (1) OOV (Out-of-Vocabulary) — words not in the vocabulary have no representation. (2) Vocabulary explosion — morphologically rich languages or domain text may need millions of word types. Subword tokenizers find a middle ground between characters and words.

A. BPE — Byte Pair Encoding (GPT, RoBERTa)

  • ▸
    Origin: Adapted from a data compression algorithm by Sennrich et al. (2016) for machine translation. Now the backbone of GPT-2, GPT-3, GPT-4, RoBERTa.
  • ▸
    Learning algorithm: Start with character vocabulary. Count all adjacent character/token pairs. Merge the most frequent pair into a new token. Repeat for a fixed number of merges (vocabulary size). The final vocabulary contains both characters and frequent subwords/words.
  • ▸
    Encoding text: Apply learned merge rules in the same order. Rare or unseen words decompose into known subwords. "unhappiness" → ["un", "happiness"] (if "happiness" was merged) or further ["un", "happy", "ness"].
BPE Learning — Step by Step
Training corpus
"low low low lower lower newest newest widest"
Initial vocab
l, o, w, e, r, n, s, t, i, d </w> (chars + end-of-word)
Merge 1 (most freq pair)
('e', 's') → 'es' (appears in "newest" ×2, "widest" ×1)
Merge 2
('es', 't') → 'est' (now "est" is a subword)
Merge 3
('l', 'o') → 'lo' (frequent in "low", "lower")
After N merges
"lower" → ["low", "er"] | "newest" → ["new", "est"] | "widest" → ["wid", "est"]

B. WordPiece (BERT)

  • ▸
    Algorithm: Similar to BPE but uses likelihood-based merging — merge the pair that maximizes the likelihood of the training data under a language model, not just raw frequency. This tends to produce more linguistically meaningful subwords.
  • ▸
    Notation convention: Tokens that are the start of a word have no prefix. Continuation tokens are prefixed with ##. "playing" → ["play", "##ing"]. This distinguishes word-initial from continuation subwords.
  • ▸
    Used in: BERT, DistilBERT, ALBERT, ELECTRA. Vocabulary size 30,000 for BERT-base English.

C. SentencePiece (T5, LLaMA, mT5)

  • ▸
    Key difference: Language-agnostic. Treats text as a raw stream of Unicode characters — no language-specific preprocessing or whitespace assumption. Works directly on raw text. Perfect for multilingual models or languages without spaces (Chinese, Japanese).
  • ▸
    Algorithm: Uses BPE or Unigram Language Model under the hood. Unigram LM: starts with a large vocabulary, iteratively prunes tokens that least reduce the likelihood of the corpus.
  • ▸
    Whitespace as character: SentencePiece uses ▁ (underscore) to represent a word-initial space, encoding position information within the token itself. "Hello world" → ["▁Hello", "▁world"].
  • ▸
    Used in: T5, mT5, LLaMA (1/2/3), Mistral, Falcon, Google's multilingual models.

Exam Point — BPE vs WordPiece vs SentencePiece

BPE: Greedy frequency-based merging. Most common. GPT family. WordPiece: Likelihood-based merging, ## prefix for continuation. BERT family. SentencePiece: Language-agnostic raw-text tokenization using ▁ for space. Used for multilingual and non-English models. All three are subword tokenizers solving the OOV problem — they differ in the merge criterion and language assumptions.

Section 5

Frequency Distribution

5.1 What is Frequency Distribution?

A frequency distribution is a count of how often each unique token appears in a corpus. It is the most fundamental descriptive statistic for text and provides the foundation for bag-of-words models, TF-IDF, n-gram models, and vocabulary construction.

Python — Frequency Distribution with NLTK
from nltk import FreqDist from nltk.tokenize import word_tokenize text = "the cat sat on the mat the cat ate the rat" tokens = word_tokenize(text) fdist = FreqDist(tokens) print(fdist.most_common(5)) # → [('the', 4), ('cat', 2), ('sat', 1), ('on', 1), ('mat', 1)] print(fdist['cat']) # → 2 print(fdist.freq('cat')) # → 0.182 (relative frequency) print(fdist.N()) # → 11 (total tokens) print(fdist.B()) # → 8 (unique token types)

5.3 Zipf's Law

A fundamental empirical law observed in all natural languages:

Zipf's Law

If words are ranked by frequency (rank 1 = most frequent), then the frequency of a word is inversely proportional to its rank:

f(word) ∝ 1/rank

The most frequent word appears ~twice as often as the 2nd most frequent, ~3× the 3rd, etc. In English: "the" (rank 1) ≈ 7% of all tokens. "of" (rank 2) ≈ 3.5%. "and" (rank 3) ≈ 2.3%.

Implication for NLP: A small number of very frequent words (stopwords) dominate raw counts. A huge number of rare words each appear very rarely. This "long tail" distribution shapes every frequency-based NLP method.

5.4 TF-IDF — Term Frequency — Inverse Document Frequency

Raw frequency counts are dominated by common words (the, a, is) that carry little discriminative information. TF-IDF weights each term by how distinctive it is across documents:

TF-IDF Formula
TF(t, d) = count(t in d) / total tokens in d ← term frequency in doc d
IDF(t, D) = log( |D| / |{d ∈ D : t ∈ d}| ) ← inverse doc frequency
TF-IDF(t,d,D) = TF(t,d) × IDF(t,D)

|D| = total documents in corpus
|{d : t ∈ d}| = number of documents containing term t

High TF-IDF: word frequent in THIS doc but rare across all docs → discriminative
Low TF-IDF: word frequent in MANY docs (stopwords) → not discriminative
  • ▸
    Intuition: "machine" appearing often in one document about engineering has high TF. If "machine" also appears in most documents in the corpus, IDF is low → TF-IDF penalizes it. "hydraulic" appearing in only 2 of 10,000 documents has high IDF → given high TF it gets high TF-IDF.
  • ▸
    Smoothing: Add 1 inside the log to avoid division by zero for terms appearing in all documents: IDF = log(|D|/(1+df)) + 1. Standard in sklearn's TfidfVectorizer.
  • ▸
    Applications: Document retrieval (return documents with highest TF-IDF score for query terms), keyword extraction (top TF-IDF words in a document), feature vectors for text classification.

Stopwords and Text Normalization

6.1 Stopwords

Stopwords are high-frequency function words (the, a, is, in, on, and, but, of...) that appear in almost every document and carry little discriminative meaning. In Bag-of-Words and TF-IDF approaches, removing them reduces vocabulary size, speeds up processing, and often improves model performance.

  • ▸
    NLTK English stopwords: 179 words including: "i, me, my, myself, we, our, you, he, she, it, the, a, an, is, are, was, were, be, been, have, has, do, does, did, will, would, can, could, should, may, might, shall, to, of, in, for, on, with, at, by, from..."
  • ▸
    When to remove: BOW/TF-IDF features for topic classification, clustering, information retrieval. Removes noise from frequency analysis.
  • ▸
    When NOT to remove: Sentiment analysis ("not good" — "not" is a stopword but critical). Any task where grammatical function words carry meaning. Deep learning models (BERT) — the model learns which words are important; don't remove anything.
  • ▸
    Custom stopwords: Domain-specific — in a medical corpus, "patient" may be so ubiquitous it's uninformative. Add domain stopwords to the standard list.
Python — Stopword Removal (NLTK)
from nltk.corpus import stopwords from nltk.tokenize import word_tokenize stop_words = set(stopwords.words('english')) tokens = word_tokenize("the quick brown fox jumps over the lazy dog") filtered = [w for w in tokens if w.lower() not in stop_words] print(filtered) # → ['quick', 'brown', 'fox', 'jumps', 'lazy', 'dog']

6.2 Case Normalization

  • ▸
    Lowercase all: Standard for most NLP tasks. "Apple", "APPLE", "apple" → same token. Reduces vocabulary size significantly.
  • ▸
    Preserve case when needed: NER (Named Entity Recognition) uses case as a signal — "Apple Inc." is an entity partly because it's capitalized. Sentiment analysis may use ALL-CAPS as a strong sentiment signal.
  • ▸
    True-case models: Models that predict the original casing given lowercase input. Used as a post-processing step in MT and ASR.
Section 7

Stemming

7.1 What is Stemming?

Stemming is a heuristic rule-based process that reduces a word to its base or root form (called the stem) by removing suffixes and prefixes. The stem need not be a real word — it is simply a common base form that groups morphologically related words together.

Purpose of Stemming

Words like "run", "running", "runs", "runner" all refer to the same core concept. Without stemming, a BOW or TF-IDF model treats them as four completely different features. Stemming collapses them to one — "run" — reducing vocabulary size and allowing the model to generalize across word forms.

7.2 Porter Stemmer (1980) — Most Widely Used

The Porter Stemmer applies a cascade of rules in 5 steps, progressively removing common English suffixes:

Porter Stemmer — Rule Steps (Simplified)
Step 1a — Plurals
SSES → SS: "caresses" → "caress" | IES → I: "ponies" → "poni" | SS → SS | S → ∅: "cats" → "cat"
Step 1b — -ed, -ing
EED → EE: "agreed" → "agree" | (v*) ED → ∅: "plastered" → "plaster" | (v*) ING → ∅: "running" → "run"
Step 2 — derivations
ATIONAL → ATE: "relational" → "relate" | IZATION → IZE: "digitization" → "digitize"
Step 3–5 — more suffix removal
ICATE → IC: "triplicate" → "triplic" | AL → ∅: "revival" → "reviv"
Python — Porter, Snowball, Lancaster Stemmers
from nltk.stem import PorterStemmer, SnowballStemmer, LancasterStemmer words = ["running", "runs", "easily", "fairly", "generously", "relational"] porter = PorterStemmer() snow = SnowballStemmer("english") lanc = LancasterStemmer() for w in words: print(f"{w:15} | Porter: {porter.stem(w):10} | Snowball: {snow.stem(w):10} | Lancaster: {lanc.stem(w)}") # running | Porter: run | Snowball: run | Lancaster: run # easily | Porter: easili | Snowball: easili | Lancaster: eas # generously | Porter: generous | Snowball: generous | Lancaster: gen # relational | Porter: relat | Snowball: relat | Lancaster: rel

7.3 Stemming Algorithms Compared

StemmerAggressivenessSpeedLanguagesKey Property
Porter (1980) Moderate Fast English only Most common, well-studied. Good balance of accuracy and simplicity. Classic 5-step rule cascade.
Porter2 / Snowball (2001) Moderate–aggressive Fast 15+ languages Improved Porter. Handles more edge cases. Multilingual via "Snowball" framework. Recommended over Porter.
Lancaster / Paice (1990) Very aggressive Fastest English only Most aggressive — can over-stem. "general" → "gen". High error rate. Rarely preferred.
Lovins (1968) Aggressive Moderate English only First published stemmer. Single-pass, largest suffix removal list.
Krovetz (1993) Light Slower English only Dictionary-based. Always produces real words. Used in IR tasks where readable output needed.

7.4 Stemming Errors

  • ▸
    Over-stemming: Different words incorrectly stem to the same base. "university" and "universe" both stem to "univers" — they are semantically very different! This conflates distinct concepts.
  • ▸
    Under-stemming: Words that should have the same base get different stems. "alumnus", "alumni", "alumna" should all stem to the same base but may not with all stemmers.
  • ▸
    Non-words: Stems are often not real English words — "easily" → "easili", "running" → "run" (ok), "relational" → "relat". This makes output hard to read or interpret for downstream tasks that need real words.
Section 8

Lemmatization

8.1 What is Lemmatization?

Lemmatization reduces a word to its canonical dictionary form — the lemma. Unlike stemming, lemmatization uses a vocabulary and morphological analysis to ensure the output is always a real, valid word. It considers the word's part of speech to determine the correct base form.

Lemmatization Examples (WordNet Lemmatizer)
Verb forms
"running" (VBG) → "run" | "ran" (VBD) → "run" | "runs" (VBZ) → "run"
Noun forms
"cats" → "cat" | "children" → "child" | "mice" → "mouse" | "feet" → "foot"
Adjective forms
"better" (JJR) → "good" | "worst" (JJS) → "bad" | "bigger" → "big"
POS-dependent
"meeting" as NOUN → "meeting" (a noun, base form is the same) | "meeting" as VERB → "meet"
Python — WordNetLemmatizer (NLTK)
from nltk.stem import WordNetLemmatizer from nltk.corpus import wordnet lem = WordNetLemmatizer() # Must provide POS tag for correct lemmatization print(lem.lemmatize("running", pos='v')) # → "run" print(lem.lemmatize("running", pos='n')) # → "running" (noun, already base form) print(lem.lemmatize("better", pos='a')) # → "good" print(lem.lemmatize("mice")) # → "mouse" (default: noun) print(lem.lemmatize("children")) # → "child" # Helper function: map NLTK POS tags to WordNet POS def get_wordnet_pos(tag): if tag.startswith('J'): return wordnet.ADJ if tag.startswith('V'): return wordnet.VERB if tag.startswith('R'): return wordnet.ADV return wordnet.NOUN # default

8.2 SpaCy Lemmatization (Modern Approach)

Python — SpaCy Lemmatization (Preferred for Production)
import spacy nlp = spacy.load("en_core_web_sm") doc = nlp("The geese were running across the frozen ponds") for token in doc: print(f"{token.text:12} | Lemma: {token.lemma_:10} | POS: {token.pos_}") # The | Lemma: the | POS: DET # geese | Lemma: goose | POS: NOUN # were | Lemma: be | POS: AUX # running | Lemma: run | POS: VERB # across | Lemma: across | POS: ADP # frozen | Lemma: freeze | POS: VERB # ponds | Lemma: pond | POS: NOUN

Exam Point — When to Use Which

Use Stemming when: Speed is critical, interpretability doesn't matter, task is information retrieval/BOW classification, working with large English corpora. Use Lemmatization when: Linguistic accuracy matters, you need real words in output (knowledge bases, chatbots), working with irregular morphology, using POS information. Use neither for deep learning: BERT/GPT handle morphology internally through attention — neither stemming nor lemmatization is applied to input text for Transformer-based models.

10.1 What is POS Tagging?

Part-of-Speech (POS) Tagging assigns a grammatical category (noun, verb, adjective, adverb, preposition, etc.) to each token in a sentence. The process disambiguates the grammatical function of each word based on its context, since many English words can serve multiple grammatical roles.

10.4 POS Tagging Approaches

  • ▸
    Rule-based: Hand-written morphological and contextual rules. Brill tagger (1992) — transformation-based learning. Starts with high-frequency tag for each word, applies ordered rules to correct errors. Fast but limited coverage.
  • ▸
    Hidden Markov Model (HMM): Models tag sequences as a Markov chain. Emission probability P(word|tag) and transition probability P(tag_i|tag_{i-1}). Viterbi decoding finds the most likely tag sequence. Accuracy ~95% on standard English but trained separately from the rest of the pipeline.
  • ▸
    Maximum Entropy Markov Model (MEMM): Log-linear model that conditions on arbitrary features (not just previous tag). Overcomes limited features of HMM. Suffers from label bias problem (solved by CRF).
  • ▸
    CRF (Conditional Random Field): Undirected graphical model. Models the joint probability of the entire tag sequence given the entire word sequence. Addresses label bias. BiLSTM-CRF was the gold standard before Transformers.
  • ▸
    Transformer-based (Current SOTA): Fine-tune BERT/RoBERTa with a token classification head. Each token's output → linear layer → softmax over POS tags. Accuracy ~97–98% on Wall Street Journal Penn Treebank. SpaCy's modern pipeline uses transformer-based taggers.
Python — POS Tagging (NLTK + SpaCy)
import nltk, spacy text = "The quick brown fox jumps over the lazy dog" # NLTK POS tagging tokens = nltk.word_tokenize(text) tags = nltk.pos_tag(tokens) # → [('The','DT'),('quick','JJ'),('brown','JJ'),('fox','NN'),('jumps','VBZ'),...] # SpaCy POS tagging (fine-grained + universal) nlp = spacy.load("en_core_web_sm") doc = nlp(text) for token in doc: print(token.text, token.pos_, token.tag_) # quick ADJ JJ ← universal + fine-grained # jumps VERB VBZ

10.5 Applications of POS Tagging

  • ▸
    Lemmatization: POS tag determines correct lemma — "better" as adjective → "good"; as verb → "better". Without POS, lemmatization is often wrong.
  • ▸
    Named Entity Recognition: Entities are typically noun phrases (NNP, NNS). POS filters candidate entity spans. Reduces search space dramatically.
  • ▸
    Dependency parsing: POS tags are features for dependency parsers — knowing "jumps" is VBZ helps predict it is the head of the verb phrase.
  • ▸
    Keyword extraction: Extract nouns and proper nouns as candidate keywords/topics from documents (noun phrases carry the content).
  • ▸
    Syntactic disambiguation: Resolve word sense ambiguity — "bank" tagged as NN (noun) vs VB (verb) changes its meaning completely. POS is a key feature for word sense disambiguation.

Bigrams, Trigrams and N-grams

11.1 What are N-grams?

An N-gram is a contiguous sequence of N items from a text. Items can be characters, words, syllables, or any other tokens. N-grams capture local context — the co-occurrence patterns of adjacent tokens — which a bag-of-words model ignores.

Why N-grams Matter

The phrase "not good" has the opposite sentiment to "good." A unigram (1-gram) bag-of-words model sees {"not", "good"} — two separate positive signals. A bigram model sees "not_good" as a single feature capturing the negation. N-grams preserve local word order and context that unigrams discard, significantly improving many NLP tasks.

11.2 N-gram Types

All N-gram Types — "I love natural language processing"
Unigrams (1-grams)
["I", "love", "natural", "language", "processing"]
Bigrams (2-grams)
["I love", "love natural", "natural language", "language processing"]
Trigrams (3-grams)
["I love natural", "love natural language", "natural language processing"]
4-grams
["I love natural language", "love natural language processing"]
Character bigrams
["I ", " l", "lo", "ov", "ve", "e ", " n", "na", "at", ...]

11.3 Generating N-grams in Python

Python — N-gram Generation (NLTK + Manual)
from nltk import bigrams, trigrams, ngrams from nltk.tokenize import word_tokenize text = "I love natural language processing" tokens = word_tokenize(text) # Bigrams list(bigrams(tokens)) # → [('I','love'), ('love','natural'), ('natural','language'), ('language','processing')] # Trigrams list(trigrams(tokens)) # → [('I','love','natural'), ('love','natural','language'), ('natural','language','processing')] # General N-grams (n=4) list(ngrams(tokens, 4)) # → [('I','love','natural','language'), ('love','natural','language','processing')] # N-gram frequency distribution from nltk import FreqDist bg_freq = FreqDist(bigrams(tokens)) bg_freq.most_common(5) # → top 5 most frequent bigrams

11.4 N-gram Count — Formula

Number of N-grams from a sequence of T tokens
Number of N-grams = T − N + 1

Example: "I love natural language processing" (T = 5 tokens)
Unigrams (N=1): 5 − 1 + 1 = 5
Bigrams (N=2): 5 − 2 + 1 = 4
Trigrams (N=3): 5 − 3 + 1 = 3
4-grams (N=4): 5 − 4 + 1 = 2
5-grams (N=5): 5 − 5 + 1 = 1

11.5 Applications of N-grams

  • ▸
    Language modeling: N-gram language models estimate P(word_n | word_{n-1}, ..., word_{n-N+1}) — probability of next word given N-1 preceding words. Foundation of early NLP before neural models. Still used in speech recognition, OCR, spelling correction.
  • ▸
    Text classification features: Adding bigrams and trigrams to TF-IDF feature vectors significantly improves sentiment analysis and topic classification — captures negation, multi-word expressions, domain jargon ("not good", "machine learning", "credit card").
  • ▸
    Spell checking: Character N-grams create a word's "fingerprint." Words with many overlapping character N-grams are likely corrections (edit distance proxy).
  • ▸
    Plagiarism detection: Documents sharing many distinctive N-grams are likely plagiarized. N-gram fingerprinting (shingling) is used by Turnitin and similar systems.
  • ▸
    Machine translation evaluation (BLEU): BLEU score measures n-gram precision of MT output against reference translations for N=1,2,3,4. Standard automatic MT evaluation metric.
  • ▸
    Collocation detection: Bigrams/trigrams with unusually high frequencies are collocations — multi-word expressions that function as a unit ("New York", "kick the bucket", "machine learning").
Section 12

N-gram Language Models

12.1 N-gram Language Model Definition

An N-gram language model estimates the probability of a sequence of words by decomposing it into conditional probabilities, where each word's probability depends only on the preceding N-1 words (Markov assumption):

N-gram Language Model — Probability Estimation
Chain rule: P(w₁w₂...wₘ) = P(w₁)·P(w₂|w₁)·P(w₃|w₁w₂)...P(wₘ|w₁...wₘ₋₁)

Bigram model (Markov order-1 approximation):
P(wₙ|wₙ₋₁) ≈ C(wₙ₋₁, wₙ) / C(wₙ₋₁)

C(wₙ₋₁, wₙ) = count of bigram "wₙ₋₁ wₙ" in corpus
C(wₙ₋₁) = count of unigram "wₙ₋₁" in corpus

Trigram model (Markov order-2):
P(wₙ|wₙ₋₂,wₙ₋₁) ≈ C(wₙ₋₂,wₙ₋₁,wₙ) / C(wₙ₋₂,wₙ₋₁)

Full sentence probability (bigram):
P(I love NLP) = P(I) × P(love|I) × P(NLP|love)

12.2 Smoothing — Handling Zero Counts

  • ▸
    The zero-count problem: If a bigram (w_{n-1}, w_n) never appeared in training data, P(w_n|w_{n-1}) = 0 → the entire sentence probability = 0. This is the data sparsity problem — most N-grams in a large vocabulary never appear in any finite training corpus.
  • ▸
    Laplace (Add-1) smoothing: Add 1 to all counts: P(wₙ|wₙ₋₁) = (C(wₙ₋₁,wₙ)+1) / (C(wₙ₋₁)+V) where V = vocabulary size. Simple but often over-smooths — transfers too much probability mass to unseen events.
  • ▸
    Add-k smoothing: Add k (0 < k < 1) instead of 1. Reduces over-smoothing. k is a tunable hyperparameter.
  • ▸
    Kneser-Ney smoothing: State-of-the-art N-gram smoothing. Uses absolute discounting + a novel continuation probability: P_KN(w|context) based on how many unique contexts w appears in (not just its raw frequency). Much better than Laplace for language modeling perplexity.
  • ▸
    Stupid Backoff: Used by Google for very large N-gram models. If C(wₙ₋₁,wₙ) = 0, back off to (N-1)-gram with a score penalty (multiply by 0.4). Simple, scalable, effective.

12.3 Perplexity — Evaluating Language Models

Perplexity — Language Model Evaluation Metric
PP(W) = P(w₁w₂...wₙ)^{-1/N}
= ⁿ√( 1 / P(w₁w₂...wₙ) )

Equivalently: PP(W) = 2^{-l} where l = (1/N) Σ log₂ P(wᵢ|context)

Lower perplexity = better language model (more confident, less "perplexed")

Example:
Uniform model (V=50,000 words): PP = 50,000 (maximally perplexed)
Bigram model on WSJ: PP ≈ 137
Trigram model on WSJ: PP ≈ 74
Large neural LM: PP < 20 (on same test set)

Named Entity Recognition (NER)

13.1 What is NER?

Named Entity Recognition (NER) is the NLP task of identifying and classifying named entities — specific real-world objects referred to by name — in text. NER is a fundamental information extraction task that assigns entity type labels to spans of text.

13.2 NER in Action — Annotated Example

13.4 BIO Tagging Scheme

NER is formulated as a sequence labeling problem. Each token receives a label encoding both whether it's part of an entity AND its position within the entity span:

BIO / IOB Tagging Scheme
B-TYPE: Beginning of an entity of TYPE ← first token of the entity
I-TYPE: Inside/continuation of entity of TYPE ← subsequent tokens
O: Outside — not part of any entity

Example: "Barack Obama visited New York City"
Barack → B-PER
Obama → I-PER
visited → O
New → B-LOC
York → I-LOC
City → I-LOC

BIOES extends BIO with:
S-TYPE: Single-token entity ("London" → S-LOC)
E-TYPE: End of multi-token entity ("Obama" → E-PER if end)

13.5 NER Approaches

A. Rule-Based NER

  • ▸
    Gazetteers (lookup lists): Maintain dictionaries of known entities — list of all countries, capital cities, company names, person names. Fast, 100% precision for known entities. Misses novel/unlisted entities.
  • ▸
    Regular expressions: Pattern-based extraction for structured entities. Dates, money values, phone numbers, IDs are well-captured by regex. Example: \$[\d,]+(?:\.\d{2})? matches money amounts.
  • ▸
    SpaCy EntityRuler: Add custom rule-based patterns alongside statistical model. Useful for domain-specific entities (drug names, product codes) that don't appear in training data.

B. Statistical NER — CRF

  • ▸
    Feature engineering for CRF: Word form, lowercase, POS tag, suffix (last 2–3 chars), prefix, is_capitalized, is_digit, is_in_gazetteer, word shape (X = capital, x = lowercase, d = digit: "Obama" → "Xxxxx"). Context features: same for surrounding ±2 tokens.
  • ▸
    CRF decoding: Viterbi algorithm finds the most likely label sequence considering all label-to-label transition scores plus per-token emission scores. Outperforms greedy classification by respecting label dependencies (I-PER cannot follow B-ORG).

C. Deep Learning NER — BiLSTM-CRF and BERT

  BERT-based NER Architecture:

  Input:  ["Barack", "Obama", "visited", "Delhi"]

  [CLS] Barack Obama visited Delhi [SEP]
    │      │      │       │       │
  BERT Encoder (12 transformer layers)
    │      │      │       │       │
  hidden states per token (768-dim each)
    │      │      │       │       │
  Linear classifier (768 → num_labels)
    │      │      │       │       │
  [ignore] B-PER  I-PER    O    B-GPE    ← predicted BIO labels

  Loss: Cross-entropy on BIO labels (ignore CLS/SEP tokens)
  Decoding: Viterbi (with CRF layer) or greedy argmax
Figure 13.1 — BERT-based NER. Each subword token gets a BIO label prediction. Multi-subword words: take the label of the first subword.
  • ▸
    BiLSTM-CRF (2015–2018 standard): Bidirectional LSTM generates context-aware token representations. CRF layer on top models label transitions. Character CNN captures morphological features (capitalization, prefixes, suffixes). ~91% F1 on CoNLL-2003 English NER.
  • ▸
    BERT fine-tuned (2018–present standard): BERT contextual embeddings encode rich bidirectional context. Linear classification head. No CRF needed (BERT's contextual representations are so rich). ~93% F1 on CoNLL-2003. Domain-specific BERT (BioBERT, LegalBERT) exceeds 90% F1 on medical/legal NER.
  • ▸
    Subword token handling: BERT splits words into subwords. For NER, only the first subword of each word contributes a label. Subsequent subwords get a special label (X or -100 in loss computation) and are ignored during evaluation.

13.6 NER Evaluation

  • ▸
    Entity-level F1 (strict): A predicted entity is correct only if BOTH the span boundaries AND the entity type match the ground truth exactly. Partial matches don't count. This is the standard CoNLL evaluation metric.
  • ▸
    Token-level F1: Evaluate BIO labels at token level. Simpler but inflated — can get partial credit for getting the boundary wrong.
  • ▸
    Macro vs Micro F1: Macro F1: average F1 equally across entity types (treats rare entities equally). Micro F1: weight by entity frequency (common entities dominate). Report both for imbalanced NER datasets.
  • ▸
    seqeval library: The standard Python library for NER evaluation. Computes precision, recall, F1 for each entity type plus overall, using entity-level strict matching.
Python — NER with SpaCy
import spacy nlp = spacy.load("en_core_web_sm") text = "Apple Inc. acquired Beats Electronics for $3 billion in Cupertino in 2014." doc = nlp(text) for ent in doc.ents: print(f"{ent.text:25} | {ent.label_:10} | start:{ent.start_char} end:{ent.end_char}") # Apple Inc. | ORG | start:0 end:10 # Beats Electronics | ORG | start:20 end:37 # $3 billion | MONEY | start:42 end:52 # Cupertino | GPE | start:56 end:65 # 2014 | DATE | start:69 end:73

Module 2 — Complete Summary

  • 1.
    Text Mining: Deriving structured knowledge from unstructured text. Pipeline: Acquisition → Pre-processing → Feature Extraction → Pattern Discovery → Evaluation. Pre-processing quality determines everything downstream.
  • 2.
    Text Cleaning: HTML strip → URL removal → encoding normalization → lowercasing → number/punctuation handling → contraction expansion → whitespace normalization. Task-dependent — no universal cleaning pipeline. Deep learning models: minimal cleaning only.
  • 3.
    Tokenization: Split text into processing units. Levels: sentence, word, subword, character, byte. Challenges: contractions, abbreviations, languages without spaces. Word tokenization → OOV problem. Subword tokenization solves OOV.
  • 4.
    Tokenizer Types: Rule-based (NLTK word_tokenize, SpaCy). Subword: BPE (greedy frequency merge, GPT), WordPiece (likelihood merge + ## prefix, BERT), SentencePiece (raw-text, ▁ space marker, T5/LLaMA). All subword tokenizers eliminate the OOV problem.
  • 5.
    Frequency Distribution: Count of each token. FreqDist(tokens). Zipf's Law: f ∝ 1/rank. TF-IDF = TF × IDF(= log(N/df)) — weights words by how distinctive they are per document. Foundation of information retrieval.
  • 6.
    Stemming: Heuristic suffix removal → stem (may not be a real word). Porter: 5-step rules. Snowball: multilingual improvement. Lancaster: most aggressive. Fast, no dictionary. Errors: over-stemming ("university"/"universe" → "univers"), under-stemming.
  • 7.
    Lemmatization: Dictionary + morphological analysis → lemma (always a real word). POS-aware: "better" (adj) → "good"; "running" (verb) → "run"; "running" (noun) → "running". Handles irregular forms: "ran"→"run", "mice"→"mouse", "children"→"child". Slower but more accurate than stemming.
  • 8.
    POS Tagging: Assign grammatical category (NN, VB, JJ, RB, DT, IN...) to each token using context. Penn Treebank: 45 tags. Approaches: Rule-based → HMM → MEMM → CRF → BiLSTM-CRF → BERT (~97–98%). Critical for lemmatization, NER, parsing, keyword extraction.
  • 9.
    N-grams: Contiguous N-token sequences. Number of N-grams = T−N+1. Bigrams capture local context (negation: "not good"). Uses: language modeling, text classification features, spell checking, plagiarism detection, BLEU evaluation. N-gram LM: P(wₙ|context) = count(N-gram)/count(context). Smoothing (Laplace, Kneser-Ney) handles zero counts.
  • 10.
    Named Entity Recognition: Identify and classify named entity spans — PER, ORG, LOC, DATE, MONEY, etc. BIO tagging: B=beginning, I=inside, O=outside entity spans. Approaches: Gazetteer/rules → CRF (hand-crafted features) → BiLSTM-CRF → BERT fine-tuned (~93% F1 CoNLL-2003). Evaluated by entity-level F1 using seqeval.
