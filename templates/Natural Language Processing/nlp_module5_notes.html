AI Chatbots — Overview and Types

1.1 What is an AI Chatbot?

An AI Chatbot is a software application designed to simulate human conversation through natural language — text or speech. It uses NLP concepts (from Modules 1–4) to understand user input, reason about it, and generate contextually appropriate responses. Modern AI chatbots range from simple rule-based FAQ answerers to sophisticated conversational agents capable of nuanced multi-turn dialogue.

Chatbot = NLP Pipeline in Conversational Form

Every concept from previous modules is directly applied: Text Cleaning & Tokenization (Module 2) preprocesses the input. POS Tagging, NER, Chunking (Modules 2, 4) extract key information. Text Classification (Module 3) identifies intent. Sequence Tagging (Module 4) extracts slot values. Language Models (Module 4) generate responses. A chatbot is the end-to-end application of all NLP components working together in real time.

1.2 Taxonomy of Chatbots

Type 1
Rule-Based Chatbots

How it works: Predefined IF-THEN rules and pattern matching. "If user says 'hello' → respond 'Hi! How can I help?'". Uses regular expressions, keyword spotting, and decision trees.

Examples: ELIZA (1966), early FAQ bots, phone IVR systems. Simple customer service bots for limited domains.

Advantages: Predictable, controllable, no training data needed, 100% interpretable. Perfect for narrow, well-defined domains with finite possible queries.

Limitation: Cannot handle paraphrasing, typos, or queries not explicitly anticipated by rule designers.
Type 2
Retrieval-Based Chatbots

How it works: Maintain a database of candidate responses. Given user input, retrieve the most semantically similar question from the database and return its paired answer. Uses TF-IDF, cosine similarity, or sentence embeddings.

Examples: FAQ chatbots, customer support bots. Amazon Kendra, IBM Watson Discovery-based bots. ChatGPT with RAG (Retrieval Augmented Generation).

Advantages: Responses are always grammatically correct (pre-written). Can be tuned for accuracy. No generation hallucination.

Limitation: Cannot respond to truly novel questions. Limited to existing knowledge base content.
Type 3
Generative Chatbots

How it works: Neural sequence-to-sequence models generate responses from scratch. Input sequence → Encoder → Decoder → Generated response. Trained on large conversation datasets (dialogue corpora).

Examples: DialoGPT, BlenderBot, ChatGPT, Llama-based assistants. Generate token by token using autoregressive language models.

Advantages: Can handle any query. Novel, contextual, flexible responses. Scales with training data and model size.

Limitation: Can hallucinate facts. Harder to control tone, accuracy, and factual grounding. Computationally expensive.
Type 4
Task-Oriented Dialogue Systems

How it works: Goal-directed conversation to complete a specific task (book a flight, order food, check weather). Uses structured NLU (intent + slots) and dialogue state tracking. Follows a conversation flow to gather all required information.

Examples: Siri, Alexa, Google Assistant, Rasa-based bots. All voice assistants are primarily task-oriented.

Advantages: High accuracy for specific tasks. Well-defined success criteria. Easier to evaluate and debug.

Best practice for enterprise chatbots: task-oriented for structured tasks + retrieval for FAQ.

2.2 Core Components Explained

Component 1
Natural Language Understanding (NLU)

Converts raw user text into structured meaning. Two primary outputs: Intent (what the user wants to do) and Entities/Slots (the specific values needed to fulfill the intent).

NLU techniques: Text classification (Naïve Bayes, BERT) for intent. Sequence labeling (BIO tagging, NER, CRF) for entities. Trained on labeled conversational data (utterance, intent, entity annotation).

Tools: Rasa NLU, Dialogflow NLU, spaCy, BERT fine-tuned on intent classification.
Component 2
Dialogue State Tracker (DST)

Maintains a running record of the conversation state — what intent is being pursued, which slots have been filled, and what has been said by each party across all turns. Also called "belief state."

State = {intent: X, slot₁: val₁, slot₂: val₂, ...}. Updated at every turn. Handles corrections ("Actually, make that Mumbai to Chennai") by overwriting previous slot values.

Implemented as: Python dictionary, database record, or neural state tracker (end-to-end trainable).
Component 3
Dialogue Policy

Decides what action to take given the current dialogue state. Rule-based policy: IF all slots filled THEN execute task. Learned policy: train using reinforcement learning or supervised imitation learning on human dialogue logs.

Actions: Ask for missing slot, Confirm understanding, Execute API call, Clarify ambiguity, Apologize and redirect, Escalate to human agent.

Rasa Core uses ML-based policies. Most production systems use hybrid rule+ML policies.
Component 4
Natural Language Generation (NLG)

Converts the system's action/response plan into natural language text. Methods: Template filling (fastest, most controlled), Retrieval (select best pre-written response), Neural generation (seq2seq, GPT — most flexible).

Template: "I found {N} flights from {origin} to {destination} on {date}." Neural: GPT-4 generates a unique, contextual response each time.

Production best practice: templates for structured info, LLM for open-ended responses.

3.1 Intent Detection

Intent detection is a text classification task that maps a user's utterance to one of the predefined intents the system supports. It is the "what does the user want?" component.

3.2 Entity Extraction / Slot Filling

Slot filling extracts specific parameter values from the utterance that are needed to fulfill the identified intent. Each intent has a set of required and optional slots.

  User: "Book me a business class flight from Mumbai to Delhi on December 25th"

  Tokenize + POS tag:
  Book/VB me/PRP a/DT business/NN class/NN flight/NN from/IN
  Mumbai/NNP to/IN Delhi/NNP on/IN December/NNP 25th/CD

  Sequence labeling with BIO (custom slot tags):
  Book        → O
  me          → O
  a           → O
  business    → B-CLASS        ← slot: class = "business"
  class       → I-CLASS
  flight      → O
  from        → O
  Mumbai      → B-FROM_CITY    ← slot: origin = "Mumbai"
  to          → O
  Delhi       → B-TO_CITY      ← slot: destination = "Delhi"
  on          → O
  December    → B-DATE         ← slot: date = "December 25th"
  25th        → I-DATE

  Extracted slots:
  {
    intent:      "BookFlight",
    origin:      "Mumbai",
    destination: "Delhi",
    date:        "December 25th",  → normalized → 2024-12-25
    class:       "Business",
    passengers:  None              ← missing → must ask
  }
Figure 3.1 — Slot filling via BIO sequence tagging. Each slot becomes a named entity type tagged with B- and I- prefixes.

3.3 Building NLU with NLP Tools

import spacy from sklearn.feature_extraction.text import TfidfVectorizer from sklearn.linear_model import LogisticRegression from sklearn.pipeline import Pipeline # ── INTENT CLASSIFICATION ────────────────────────────────────────── training_data = [ ("book a flight to Mumbai", "BookFlight"), ("I need a ticket to Delhi", "BookFlight"), ("find me flights tomorrow", "BookFlight"), ("what is the weather today", "CheckWeather"), ("will it rain this evening", "CheckWeather"), ("cancel my booking", "CancelOrder"), ("I want to cancel my order", "CancelOrder"), ("hello", "Greeting"), ("hi good morning", "Greeting"), ("help me", "GetHelp"), ("what can you do", "GetHelp"), ("goodbye thanks", "Goodbye"), ] utterances, intents = zip(*training_data) intent_classifier = Pipeline([ ('tfidf', TfidfVectorizer(ngram_range=(1,2), max_features=5000)), ('clf', LogisticRegression(C=5.0, max_iter=500)), ]) intent_classifier.fit(utterances, intents) def predict_intent(text): intent = intent_classifier.predict([text])[0] proba = intent_classifier.predict_proba([text])[0] confidence = max(proba) return intent, confidence # ── ENTITY / SLOT EXTRACTION using spaCy NER ────────────────────── nlp = spacy.load('en_core_web_sm') def extract_entities(text): doc = nlp(text) entities = {} for ent in doc.ents: entities[ent.label_] = ent.text return entities # ── COMBINED NLU FUNCTION ───────────────────────────────────────── def nlu_parse(user_input): intent, confidence = predict_intent(user_input) entities = extract_entities(user_input) return { "text": user_input, "intent": intent, "confidence": confidence, "entities": entities } # Test result = nlu_parse("Book a flight from Mumbai to Delhi on December 25th") print(result) # → {'intent': 'BookFlight', 'confidence': 0.94, # 'entities': {'GPE': 'Delhi', 'DATE': 'December 25th'}}
Section 4

Dialogue Management

4.1 Dialogue State Tracking

The dialogue state tracker (DST) maintains the current "belief state" — a structured representation of everything the system knows about the conversation so far, including identified intents, filled slots, and missing information.

Python — Dialogue State Tracker
class DialogueStateTracker: def __init__(self): self.state = { 'intent': None, 'origin': None, 'destination': None, 'date': None, 'class': None, 'confirmed': False, 'turn': 0, } self.history = [] def update(self, nlu_result): self.state['turn'] += 1 # Update intent if high confidence if nlu_result['confidence'] > 0.6: self.state['intent'] = nlu_result['intent'] # Update slots from entities entities = nlu_result.get('entities', {}) if 'FROM_CITY' in entities: self.state['origin'] = entities['FROM_CITY'] if 'TO_CITY' in entities: self.state['destination'] = entities['TO_CITY'] if 'DATE' in entities: self.state['date'] = entities['DATE'] if 'CLASS' in entities: self.state['class'] = entities['CLASS'] # Keep conversation history self.history.append(nlu_result['text']) return self.state def missing_slots(self): required = ['origin', 'destination', 'date'] return [s for s in required if self.state[s] is None] def is_complete(self): return len(self.missing_slots()) == 0

4.2 Dialogue Policy

The dialogue policy maps dialogue states to system actions. A rule-based policy is most interpretable and predictable for task-oriented chatbots:

Python — Rule-Based Dialogue Policy
class DialoguePolicy: def decide(self, state, missing): intent = state['intent'] # Handle greetings if intent == 'Greeting': return ('greet', {}) # Handle help if intent == 'GetHelp': return ('provide_help', {}) # Handle goodbye if intent == 'Goodbye': return ('farewell', {}) # Handle BookFlight intent if intent == 'BookFlight': if missing: return ('ask_slot', {'slot': missing[0]}) # ask first missing elif not state['confirmed']: return ('confirm', state) # confirm before booking else: return ('execute_booking', state) # do the booking # Fallback return ('fallback', {}) # Slot-to-question mapping SLOT_QUESTIONS = { 'origin': "Which city are you flying from?", 'destination': "Which city would you like to fly to?", 'date': "What date would you like to travel?", 'class': "Which class? Economy, Business, or First Class?", }
Section 5

NLG — Response Generation

5.1 Response Generation Methods

Method 1
Template-Based NLG

Pre-written response templates with placeholder slots filled from the dialogue state. Fast, predictable, grammatically correct. Best for structured task-oriented responses.

Example: "I found {count} flights from {origin} to {destination} on {date}. Prices start from ₹{price}."

Pros: Fully controlled, no hallucination. Cons: Rigid, limited variety, requires templates for every scenario.
Method 2
Retrieval-Based NLG

Retrieve the best matching pre-written response from a response library based on TF-IDF or embedding similarity to the current context. Ensures quality, human-written language.

Implementation: TfidfVectorizer on response candidates → cosine_similarity → pick top-1 response. Or: SBERT embeddings for semantic matching.

Used for FAQ responses, chitchat, and general queries. Combine with templates for task responses.
Method 3
Neural NLG (LLM-Based)

Use a pre-trained language model (GPT, T5, LLaMA) to generate responses. Provide context as a prompt: system description + conversation history + current task state → LLM generates contextual response.

Prompt: "You are a helpful flight booking assistant. User wants to book: [state]. Previous turns: [history]. Respond naturally and ask for: [missing slot]."

State-of-art for open-domain chitchat. Use sparingly for task-oriented (cost + latency). Hallucination risk.
Python — NLG Response Engine (Template + Retrieval)
from sklearn.metrics.pairwise import cosine_similarity from sklearn.feature_extraction.text import TfidfVectorizer import numpy as np # ── Template-Based NLG ──────────────────────────────────────────── TEMPLATES = { 'greet': "Hello! I'm your AI travel assistant. How can I help you today?", 'farewell': "Thank you for using our service. Have a safe journey!", 'provide_help': "I can help you: Book flights, Check weather, Cancel bookings. What would you like?", 'ask_slot': lambda params: SLOT_QUESTIONS[params['slot']], 'confirm': lambda state: ( f"To confirm: {state['origin']} → {state['destination']} on {state['date']}." f" Class: {state['class'] or 'Economy'}. Shall I proceed? (yes/no)" ), 'execute_booking': lambda state: ( f"✅ Booking confirmed! {state['origin']} to {state['destination']} on {state['date']}." f" Your reference: FL{np.random.randint(10000,99999)}." ), 'fallback': "I'm sorry, I didn't understand that. Could you rephrase?", } def generate_response(action, params): template = TEMPLATES.get(action, TEMPLATES['fallback']) if callable(template): return template(params) return template # ── Retrieval-Based NLG for FAQ ─────────────────────────────────── faq_pairs = [ ("what is the baggage limit", "Economy allows 15kg checked and 7kg cabin baggage."), ("can I change my flight date", "Yes, date changes are allowed up to 24 hours before departure for a fee."), ("how do I get a refund", "Refunds are processed in 7-10 business days to your original payment method."), ("what documents do I need", "Carry a valid government-issued photo ID. For international travel, a passport is required."), ("is food available on the flight", "Complimentary meals are served on flights over 2 hours. Snacks available on shorter flights."), ] faq_questions, faq_answers = zip(*faq_pairs) faq_vectorizer = TfidfVectorizer().fit(faq_questions) def retrieve_faq_answer(query, threshold=0.3): q_vec = faq_vectorizer.transform([query]) faq_v = faq_vectorizer.transform(faq_questions) scores = cosine_similarity(q_vec, faq_v)[0] best = np.argmax(scores) if scores[best] >= threshold: return faq_answers[best] return None # No good match — fall back to bot response

7.1 Full ML Chatbot — Integrated System

Python — Complete ML-Based Task-Oriented Chatbot
import re import numpy as np from sklearn.feature_extraction.text import TfidfVectorizer from sklearn.linear_model import LogisticRegression from sklearn.pipeline import Pipeline from sklearn.metrics.pairwise import cosine_similarity # ───────────────────────────────────────────────────────────────── # TRAINING DATA # ───────────────────────────────────────────────────────────────── intent_data = [ ("book a flight", "BookFlight"), ("I need to fly to Chennai", "BookFlight"), ("find me tickets from Kolkata", "BookFlight"), ("what is the weather in Bangalore", "CheckWeather"), ("will it rain tomorrow", "CheckWeather"), ("cancel my reservation", "CancelOrder"), ("I want a refund", "CancelOrder"), ("hello good morning", "Greeting"), ("hi there", "Greeting"), ("help me please", "GetHelp"), ("what can you do", "GetHelp"), ("goodbye thanks", "Goodbye"), ("see you later", "Goodbye"), ] texts, labels = zip(*intent_data) # ───────────────────────────────────────────────────────────────── # NLU — INTENT CLASSIFIER # ───────────────────────────────────────────────────────────────── clf = Pipeline([ ('tfidf', TfidfVectorizer(ngram_range=(1,2))), ('clf', LogisticRegression(C=5, max_iter=500)), ]) clf.fit(texts, labels) # ───────────────────────────────────────────────────────────────── # SIMPLE ENTITY EXTRACTOR (regex-based) # ───────────────────────────────────────────────────────────────── CITY_PATTERNS = ['mumbai','delhi','chennai','kolkata','bangalore','hyderabad','pune'] DATE_PATTERN = re.compile(r'\b(\d{1,2}(?:st|nd|rd|th)?\s+\w+|\btomorrow\b|\bnext \w+)') CLASS_PATTERN = re.compile(r'\b(economy|business|first class?)\b', re.I) def extract_entities(text): text_low = text.lower() found_cities = [c for c in CITY_PATTERNS if c in text_low] date_match = DATE_PATTERN.search(text_low) class_match = CLASS_PATTERN.search(text) entities = {} if len(found_cities) >= 2: entities['FROM_CITY'] = found_cities[0].title() entities['TO_CITY'] = found_cities[1].title() elif len(found_cities) == 1: entities['TO_CITY'] = found_cities[0].title() if date_match: entities['DATE'] = date_match.group(1) if class_match: entities['CLASS'] = class_match.group(1).title() return entities # ───────────────────────────────────────────────────────────────── # MAIN CHATBOT LOOP # ───────────────────────────────────────────────────────────────── class MLChatbot: def __init__(self): self.state = {'intent':None,'FROM_CITY':None,'TO_CITY':None,'DATE':None,'CLASS':None} def respond(self, user_text): # 1. NLU intent = clf.predict([user_text])[0] confidence= max(clf.predict_proba([user_text])[0]) entities = extract_entities(user_text) # 2. Update state if confidence > 0.5: self.state['intent'] = intent self.state.update({k: v for k, v in entities.items() if v}) # 3. Policy + Response if intent == 'Greeting': return "Hi! I can help you book flights, check weather, or manage bookings. What do you need?" elif intent == 'Goodbye': return "Goodbye! Have a safe journey! ✈️" elif intent == 'BookFlight': if not self.state['FROM_CITY']: return "Which city are you flying FROM?" if not self.state['TO_CITY']: return "Which city are you flying TO?" if not self.state['DATE']: return "What date would you like to travel?" return (f"✅ Booking: {self.state['FROM_CITY']} → {self.state['TO_CITY']}" f" on {self.state['DATE']}. Reference: FL{np.random.randint(10000,99999)}") elif intent == 'CheckWeather': return "Weather service: Currently 28°C and clear skies. (Live data integration needed)" elif intent == 'CancelOrder': return "Please provide your booking reference number and I'll process the cancellation." else: return "I didn't understand that. Could you rephrase? Type 'help' for options." # Run the chatbot bot = MLChatbot() print("ML Chatbot Ready! (type 'quit' to exit)\n") while True: user_in = input("You: ") if user_in.lower() == 'quit': break print(f"Bot: {bot.respond(user_in)}\n")
Exam Point — Chatbot Using All NLP Modules

Module 2 (Preprocessing): Tokenization, lowercasing, noise removal applied to user input before NLU.

Module 3 (Text Classification): TF-IDF + Logistic Regression for intent detection. Naïve Bayes is also commonly used.

Module 4 (Sequence Tagging + LM): BIO sequence labeling for slot/entity extraction. Language models for response generation.

Module 3 (Confusion Matrix): Evaluate intent classifier with P/R/F1 per intent. NER evaluated with entity-level F1.

Recommendation Engine — Overview

8.1 What is a Recommendation Engine?

A Recommendation Engine (Recommender System) is an information filtering system that predicts what items a user is likely to prefer based on their past behavior, preferences, and/or item characteristics. It reduces information overload by surfacing the most relevant items from a vast catalog.

Why Recommendation Engines Matter

Netflix attributes 80% of content watched to recommendations. Amazon reports 35% of revenue driven by its recommender. YouTube: 70% of watch time from recommendations. Spotify Discover Weekly reaches 40M users weekly. Recommendation engines are among the highest business-value ML applications ever built.

Section 9

Collaborative Filtering

9.1 User-Based Collaborative Filtering

User-based CF: Find users most similar to the target user → recommend items those similar users liked but the target user hasn't seen yet.

Cosine Similarity Between Users
sim(u, v) = cos(θ) = (u⃗ · v⃗) / (‖u⃗‖ · ‖v⃗‖)

= Σᵢ(rᵤᵢ · rᵥᵢ) / √(Σᵢ rᵤᵢ²) · √(Σᵢ rᵥᵢ²)

rᵤᵢ = rating of user u for item i (0 if unrated)

Pearson Correlation (better, handles rating scale bias):
sim(u, v) = Σᵢ(rᵤᵢ−r̄ᵤ)(rᵥᵢ−r̄ᵥ) / [√Σᵢ(rᵤᵢ−r̄ᵤ)² · √Σᵢ(rᵥᵢ−r̄ᵥ)²]

r̄ᵤ = mean rating of user u ← centers user's ratings, removes bias

Prediction of user u's rating for item i:
r̂ᵤᵢ = r̄ᵤ + (Σ_{v∈N(u)} sim(u,v) · (rᵥᵢ − r̄ᵥ)) / Σ_{v∈N(u)} |sim(u,v)|
N(u) = k nearest neighbors of user u who have rated item i

9.2 Item-Based Collaborative Filtering

Item-based CF: Find items most similar to items the target user already rated highly → recommend the most similar unseen items. More stable than user-based (item relationships change less than user behavior).

Item-Based CF — Worked Example (Predict Alice's Avengers rating)
Alice's ratings
Inception:5, The Matrix:4, Interstellar:5, Titanic:2 (Avengers:? → predict)
Avengers sim(Inception)
cosine(Avengers vector, Inception vector) = 0.87 ← both sci-fi action
Avengers sim(Matrix)
cosine(Avengers, Matrix) = 0.82
Avengers sim(Interstellar)
cosine(Avengers, Interstellar) = 0.79
Avengers sim(Titanic)
cosine(Avengers, Titanic) = 0.15 ← different genre
Predicted rating
(5×0.87 + 4×0.82 + 5×0.79 + 2×0.15) / (0.87+0.82+0.79+0.15) = 14.07 / 2.63 = 4.35
Recommendation
Predicted 4.35/5 → Recommend Avengers to Alice ✓

9.3 Matrix Factorization — SVD (Netflix Prize Winner)

Decompose the user-item rating matrix R into latent factor matrices. Each user and item is represented as a vector in a latent feature space (e.g., "action", "drama", "sci-fi" preferences). The predicted rating is the dot product of user and item latent vectors.

Matrix Factorization — SVD / ALS
R ≈ U × Σ × Vᵀ = P × Qᵀ

R ∈ ℝ^{M×N} M=users, N=items
P ∈ ℝ^{M×k} user latent factor matrix (k=50–200 factors)
Q ∈ ℝ^{N×k} item latent factor matrix

Predicted rating: r̂ᵤᵢ = pᵤ · qᵢᵀ = Σₖ pᵤₖ · qᵢₖ

Objective (minimize regularized squared error):
L = Σ_{(u,i)∈known} (rᵤᵢ − pᵤqᵢᵀ)² + λ(‖pᵤ‖² + ‖qᵢ‖²)

With biases:
r̂ᵤᵢ = μ + bᵤ + bᵢ + pᵤqᵢᵀ
μ = global average, bᵤ = user bias, bᵢ = item bias

Solved by SGD or ALS (Alternating Least Squares)
Python — Collaborative Filtering (User-Based + Matrix Factorization)
import numpy as np from sklearn.metrics.pairwise import cosine_similarity # ── Rating matrix (rows=users, cols=items, 0=unrated) R = np.array([ [5, 4, 5, 2, 0], # Alice: Inception,Matrix,Interstellar,Titanic,Avengers [4, 5, 0, 1, 4], # Bob [0, 5, 4, 0, 5], # Carol [4, 0, 5, 1, 0], # Dave [0, 4, 0, 5, 3], # Eve ], dtype=float) MOVIES = ['Inception','The Matrix','Interstellar','Titanic','Avengers'] # ── USER-BASED COLLABORATIVE FILTERING ──────────────────────────── def user_based_cf(R, user_idx, top_k=5): # Compute user-user similarity matrix sim = cosine_similarity(R) user_sim = sim[user_idx].copy() user_sim[user_idx] = 0 # exclude self scores = np.zeros(R.shape[1]) sim_sum= np.zeros(R.shape[1]) for v, s in enumerate(user_sim): if s > 0: scores += s * R[v] sim_sum += s * (R[v] > 0).astype(float) with np.errstate(invalid='ignore'): predicted = np.where(sim_sum > 0, scores/sim_sum, 0) # Only recommend items not yet rated by this user unrated = np.where(R[user_idx] == 0)[0] top_recs= sorted(unrated, key=lambda i: predicted[i], reverse=True)[:top_k] return [(MOVIES[i], round(predicted[i], 2)) for i in top_recs] print("Alice's recommendations:", user_based_cf(R, 0)) # → [('Avengers', 4.35)] ← predicted 4.35 stars for Alice # ── MATRIX FACTORIZATION with surprise library ──────────────────── from surprise import SVD, Dataset, Reader from surprise.model_selection import cross_validate import pandas as pd ratings_data = [ ('Alice','Inception',5), ('Alice','Matrix',4), ('Bob','Matrix',5), ('Bob','Inception',4), ('Carol','Matrix',5), ('Dave','Inception',4), ] df = pd.DataFrame(ratings_data, columns=['user','item','rating']) reader = Reader(rating_scale=(1,5)) data = Dataset.load_from_df(df[['user','item','rating']], reader) model = SVD(n_factors=50, n_epochs=20, lr_all=0.005, reg_all=0.02) results = cross_validate(model, data, measures=['RMSE','MAE'], cv=3, verbose=True)

9.4 Cold Start Problem

  • ▸
    New user cold start: A brand-new user has no rating history — cannot compute similarity to other users. Solutions: Ask for explicit preferences during onboarding, use demographic information, recommend globally popular items, apply knowledge-based filtering.
  • ▸
    New item cold start: A new item has no ratings — never appears in collaborative recommendations. Solutions: Content-based filtering using item features (no ratings needed), hybrid approaches, editorial curation for launch period.
  • ▸
    Sparsity problem: Large platforms have millions of users and items — most user-item combinations are unrated (>99% missing). Matrix factorization handles sparsity better than neighborhood methods by learning dense latent representations.
Section 10

Content-Based Filtering

10.1 Content-Based Filtering — NLP-Powered

Content-based filtering recommends items similar to what the user already likes, based on item features (genre, description, tags, metadata). NLP is central here — item descriptions and reviews are analyzed using TF-IDF, word embeddings, or BERT to compute semantic similarity between items.

Content-Based Similarity
1. Represent each item as a feature vector:
- Structured: genre one-hot + year + director_id + avg_rating
- NLP: TF-IDF on description + tags + reviews

2. Build user profile from rated items:
user_profile(u) = Σᵢ rᵤᵢ · item_vector(i) / Σᵢ rᵤᵢ
Weighted average of item vectors by user's ratings

3. Score unrated items against user profile:
score(u, i) = cosine(user_profile(u), item_vector(i))

4. Recommend top-k items with highest score
Python — Content-Based Movie Recommender (TF-IDF on Descriptions)

N×N item similarity matrix # ── Content-Based Recommendations for a movie ───────────────────── def recommend_similar(movie_title, top_n=3): idx = movies[movies['title'] == movie_title].index[0] scores = list(enumerate(sim_m[idx])) scores = sorted(scores, key=lambda x: x[1], reverse=True)[1:top_n+1] return [(movies.iloc[i]['title'], round(s, 3)) for i, s in scores] print(recommend_similar("Inception")) # Hybrid Recommendation Systems

11.1 Why Hybrid?

No single recommendation approach works best in all scenarios. Collaborative filtering suffers from cold start and sparsity. Content-based lacks serendipity (only recommends what's similar to what you know). Hybrid systems combine both to get the advantages of each while mitigating their weaknesses.

MethodData NeededCold StartSerendipityScalabilityNLP Role
User-Based CFUser ratingsPoor (both)HighPoor (O(U²))None
Item-Based CFUser ratingsPoor (item)MediumGood (offline)None
Matrix FactorizationUser ratingsPoor (new user)HighExcellentNone (ratings only)
Content-BasedItem featuresExcellentLowGoodCore (TF-IDF, embeddings on descriptions)
HybridBothGoodHighGoodContent vectors + CF predictions blended

11.2 Hybrid Combination Strategies

Strategy 1
Weighted Combination

Compute scores from multiple methods independently, then combine with learned or fixed weights:

score = α × CF_score + (1−α) × CB_score

α = 0.7 gives CF more weight when interaction data is dense; α = 0.3 prefers content-based for new users. α can be learned or set manually per user segment.

Simplest hybrid. Netflix uses α per user based on their interaction history density.
Strategy 2
Switching / Cascade

Switching: If user has < N ratings → use content-based (cold start). If user has ≥ N ratings → use collaborative filtering (rich data).

Cascade: Use CF to generate a candidate set of 100 items → use content-based to re-rank the top 10 using semantic similarity to user's history.

Cold start threshold N typically 10–20 ratings. Cascade is standard in industry (two-stage retrieval + ranking).
Strategy 3
Feature Augmentation

Augment CF user/item representations with NLP content features. User vector = [CF latent factors + TF-IDF of reviewed text + demographic features]. Item vector = [CF latent factors + content TF-IDF + metadata].

Train a single model (LightFM, deep learning) on this rich feature set. Naturally handles cold start (can fall back to content features when CF features are missing).

LightFM: open-source library combining CF + content features in a unified model.

NLP-Based Recommendation

12.1 Leveraging NLP for Richer Recommendations

NLP unlocks recommendation signals beyond explicit ratings — product reviews, item descriptions, social media posts, and conversational context can all be analyzed with NLP to improve recommendation quality.

NLP Technique 1
Review Sentiment for Preference Learning

Instead of only using the star rating (1–5), analyze the text of the review. A 3-star review saying "great plot but poor sound" reveals preferences the number alone hides. Extract aspect-level sentiments (Module 3 sentiment analysis) per feature (sound, plot, acting).

Build richer user profiles: user has strong positive sentiment for "plot" and "acting" across all 4-star+ reviews → recommend movies described as having "compelling plot" and "outstanding performances" in their descriptions.

Tools: VADER, BERT-based sentiment on aspects. Amazon's review-based recommendation uses this.
NLP Technique 2
Description Embeddings

Convert item descriptions to dense semantic vectors using Sentence-BERT (SBERT), Doc2Vec, or fine-tuned BERT. These capture semantic meaning better than TF-IDF — "thriller" and "suspenseful" are near in embedding space.

Compute item similarity in the embedding space using cosine similarity. Recommendations: recommend items whose SBERT-embedded description is closest to the embedding of items the user liked.

SBERT (all-MiniLM-L6-v2): 384-dim embeddings, fast inference, excellent semantic quality. Better than TF-IDF for short descriptions.
NLP Technique 3
Conversational Recommendations

Combine the chatbot (Part 1) with the recommender (Part 2). User: "I'm looking for something like Inception but more romantic." → NLU extracts: reference_movie=Inception, preferred_modification=more_romantic → Recommendation engine finds movies close to Inception in sci-fi space but with romance elements.

This is the state-of-the-art paradigm: Conversational Recommendation Systems. Used by Spotify, Netflix, and modern shopping assistants.

The full integration — this is what Section 13 builds.

14.2 Evaluating Recommendation Engines

Recommendation System Evaluation Metrics
RMSE (Root Mean Square Error): √(Σ(rᵤᵢ − r̂ᵤᵢ)² / N)
← How far are predicted ratings from actual? Lower = better.

MAE (Mean Absolute Error): (1/N) Σ |rᵤᵢ − r̂ᵤᵢ|
← Mean absolute rating prediction error. More interpretable than RMSE.

Precision@K: |relevant items in top-K| / K
← Of the K recommendations shown, how many were relevant?

Recall@K: |relevant items in top-K| / |all relevant items|
← Of all items the user would like, how many did we surface in top-K?

NDCG@K (Normalized Discounted Cumulative Gain):
DCG@K = Σᵢ₌₁ᴷ (2^relᵢ − 1) / log₂(i+1) NDCG@K = DCG@K / IDCG@K
← Position-aware: highly relevant item at position 1 better than at K

Coverage: % of catalog items ever recommended → diversity measure
Novelty: average popularity^{-1} of recommended items
Serendipity: how surprising AND relevant the recommendations are

14.4 Advanced Topics and Future Directions

Advanced Topic 1
Large Language Model Chatbots

Fine-tune LLaMA/Mistral/GPT on domain-specific conversations. Use prompt engineering with chain-of-thought reasoning. RAG (Retrieval Augmented Generation) grounds LLM responses in a knowledge base. RLHF aligns the chatbot with human preferences (Module 4 DLT concepts directly apply).

Tools: LangChain, LlamaIndex, Hugging Face TRL for fine-tuning, Pinecone/Chroma for vector DB.
Advanced Topic 2
Deep Learning Recommenders

Neural Collaborative Filtering (NCF): replace dot product with MLP. Wide & Deep (Google): memorization (wide) + generalization (deep). YouTubeNet: two-stage DNN for YouTube recommendations. BERT4Rec: model sequential user behavior with Transformers.

BERT4Rec treats user interaction history as a sequence → apply masked language model to predict next interaction.
Advanced Topic 3
Conversational Recommender Systems

Combine dialogue and recommendation in a single end-to-end system. The chatbot gathers user preferences through natural conversation and provides increasingly refined recommendations based on multi-turn feedback. Handles: "Not that genre", "something shorter", "something from 2020 onward."

Research: KGSF (Knowledge Graph Sequential Filtering), CR-Walker, C2-CRS — all combine dialogue + recommendation.
Advanced Topic 4
Ethical Considerations

Filter bubble: Recommenders amplify existing preferences → users see only content confirming their views. Solution: intentional diversity in recommendations, serendipity measures.

Chatbot bias: If trained on biased data, chatbot perpetuates stereotypes. Solution: Careful data curation, bias testing, RLHF with diverse feedback providers.

Privacy: Behavioral data collection for recommendations is privacy-sensitive. Differential privacy for recommendation training.

Module 5 — Complete Summary

  • 1.
    AI Chatbots: 4 types — Rule-Based (pattern matching, NLTK), Retrieval-Based (TF-IDF similarity), Generative (seq2seq, GPT), Task-Oriented (intent+slot+dialogue management). Every chatbot applies all previous NLP modules: tokenization, POS, NER, classification, sequence tagging, language modeling.
  • 2.
    Chatbot Architecture: 3 layers — NLU (intent classification + entity extraction), Dialogue Management (state tracker + policy), NLG (template/retrieval/neural generation). Intent = what user wants; Entities/Slots = parameter values for fulfilling the intent.
  • 3.
    NLU Implementation: Intent: TF-IDF + Logistic Regression (or BERT fine-tuned) on labeled utterance-intent pairs. Entities: BIO sequence tagging (CRF, BiLSTM+CRF, BERT), spaCy NER, or regex for structured patterns. Confidence threshold for fallback.
  • 4.
    Dialogue Management: State = Python dict of {intent, slots}. Policy = IF-THEN rules (production) or ML policy (Rasa Core). Actions: ask missing slot, confirm, execute, fallback, handoff to human.
  • 5.
    NLG: Template filling (fast, controlled, no hallucination), Retrieval (TF-IDF cosine similarity on FAQ pairs), Neural (LLM with system prompt + conversation history as context).
  • 6.
    Recommendation Engine Types: Collaborative Filtering (user-based: find similar users; item-based: find similar items; MF/SVD: latent factors), Content-Based (TF-IDF/SBERT on item descriptions), Hybrid (weighted blend, switching on cold start, cascade two-stage).
  • 7.
    Collaborative Filtering: User-item rating matrix R. User-based: sim(u,v) = cosine(u,v); predict r̂ᵤᵢ via weighted neighbor ratings. Matrix Factorization: R≈PQᵀ; minimize ‖R−PQᵀ‖² + λ regularization via SGD/ALS. Cold start problem: new user/item has no data → content-based fallback.
  • 8.
    Content-Based (NLP-Powered): TF-IDF or SBERT embeddings on item descriptions. User profile = weighted average of liked item vectors. Score = cosine(user_profile, item_vector). SBERT captures semantic similarity better than TF-IDF ("thriller" ≈ "suspenseful").
  • 9.
    Integration: Chatbot extracts user preferences via NLU → recommender surfaces relevant items → NLG presents recommendations conversationally. This is the Conversational Recommendation System paradigm used by Spotify, Netflix, Amazon.
  • 10.
    Evaluation: Chatbot: Intent Accuracy, Slot F1, Task Success Rate, Fallback Rate, User Satisfaction. Recommender: RMSE/MAE (rating prediction), Precision@K, Recall@K, NDCG@K (ranking quality), Coverage, Novelty, Serendipity.
