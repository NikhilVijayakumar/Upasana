Section 1

What are Hyperparameters?

1.1 Parameters vs Hyperparameters

In a neural network, there are two categories of values that control behavior. Understanding the difference is fundamental.

AspectParametersHyperparameters
Definition Internal values learned by the model during training External settings chosen by the user before training begins
Examples Weights W, Biases b Learning rate α, number of layers, batch size, epochs
Who sets them? Automatically updated by backpropagation + gradient descent Set manually by the engineer/researcher
How optimized? Optimized via gradient descent to minimize cost J Tuned via search strategies (grid search, random search, etc.)
Saved in model? Yes — define the trained model Recorded separately in experiment logs
Key Insight

Parameters are what the network learns. Hyperparameters are what you decide. Choosing the right hyperparameters can mean the difference between a model that barely learns and one that reaches state-of-the-art performance.

1.2 Why Hyperparameter Tuning Matters for Convergence

A deep network's ability to converge to a good solution depends critically on its hyperparameters. Poor choices lead to:

  • ▸
    Non-convergence: The cost function never decreases (usually due to learning rate too high or poor weight initialization).
  • ▸
    Very slow convergence: Training takes impractically long (learning rate too low, batch size too large).
  • ▸
    Overfitting: Network memorizes training data, performs poorly on test data (too many layers/neurons, insufficient regularization).
  • ▸
    Underfitting: Network is too simple to capture the patterns in data (too few layers, high regularization).
  • ▸
    Training instability: Loss oscillates wildly or explodes to NaN (learning rate + batch size interaction).
Section 2

Key Hyperparameters in Deep Networks

2.1 Learning Rate (α) — The Most Important Hyperparameter

  • ▸
    Definition: Controls how large a step we take along the negative gradient direction during each parameter update: W := W − α · ∂J/∂W.
  • ▸
    Typical range: 0.1, 0.01, 0.001, 0.0001. Usually searched on a logarithmic scale.
  • ▸
    Too high: Overshoots the minimum. Loss oscillates or diverges. Unstable training.
  • ▸
    Too low: Extremely slow convergence. May plateau in a flat region or saddle point.
  • ▸
    Learning Rate Schedule: Start with a larger α and decrease over time. Three common schedules: Step Decay (halve α every N epochs), Exponential Decay (α = α₀ · e^(−kt)), Cosine Annealing (smooth periodic decay).
  • ▸
    Warm-up: Modern practice — start with a very small α, ramp it up for the first few epochs, then decay. Helps with training stability in very deep networks and Transformers.

2.2 Number of Hidden Layers (Depth)

  • ▸
    Role: Determines the depth of feature hierarchy the network can learn. More layers → more abstract representations.
  • ▸
    Practical start: Start with 1–2 hidden layers and add more only if the model underfits.
  • ▸
    Deeper is not always better: Beyond a certain depth, adding layers gives diminishing returns and introduces vanishing gradient problems (mitigated by ReLU, batch normalization, residual connections).
  • ▸
    Architecture search: Choosing depth is itself an active research area (Neural Architecture Search / NAS).

2.3 Number of Neurons per Layer (Width)

  • ▸
    Role: Determines the representational capacity at each level. More neurons → richer representations per layer.
  • ▸
    Common practice: Use powers of 2 (32, 64, 128, 256, 512) for computational efficiency on GPU/CPU hardware.
  • ▸
    Funnel architecture: Decreasing width across layers (e.g. 512→256→128) forces the network to compress information — useful for autoencoders and dimensionality reduction.
  • ▸
    Same width across layers: Often used in practice as a simple starting point, tuned later.

2.4 Batch Size

  • ▸
    Definition: Number of training examples used per gradient update step.
  • ▸
    Small batches (8–64): Noisy gradient estimates, but noise acts as regularization. Better generalization, slower per-epoch time. Good for small GPUs.
  • ▸
    Large batches (256–2048+): More accurate gradient estimates, parallelizes well on large GPUs. But tends to converge to "sharp" minima with worse generalization.
  • ▸
    The "batch size sweet spot": 32–256 for most deep learning tasks. When increasing batch size, also increase learning rate proportionally (linear scaling rule).
  • ▸
    Interaction with learning rate: Larger batch → more accurate gradient → can use larger α. Relationship: αₙₑw = αₒₗd × (new_batch_size / old_batch_size).

2.5 Number of Epochs

  • ▸
    Definition: One epoch = one complete pass through the entire training dataset.
  • ▸
    Too few epochs: Model hasn't learned enough — underfitting.
  • ▸
    Too many epochs: Model memorizes training data — overfitting.
  • ▸
    Early Stopping: Monitor validation loss; stop training when validation loss starts increasing even as training loss decreases. One of the best regularization techniques.

2.6 Regularization Hyperparameters

  • ▸
    L2 regularization (λ): Adds λ · ‖W‖² to the cost. Penalizes large weights, prevents overfitting. Common values: 0.1, 0.01, 0.001.
  • ▸
    Dropout rate (p): Probability of randomly dropping a neuron during training. Typical: 0.2–0.5. Acts as ensemble of smaller networks.
  • ▸
    Batch normalization momentum: Controls how quickly the running statistics (mean, variance) update. Usually set to 0.99.

2.7 Weight Initialization

Although often set automatically, weight initialization is a critical hyperparameter that affects whether training even starts properly.

  • ▸
    Zero initialization — NEVER do this: All neurons in a layer compute identical outputs. All gradients are identical. Network cannot learn (symmetry breaking problem).
  • ▸
    Random initialization (small values): W ~ N(0, 0.01). Works for shallow networks but causes vanishing gradients for deep ones.
  • ▸
    Xavier (Glorot) initialization: W ~ N(0, 1/n[l-1]). Designed for Tanh/Sigmoid activations. Keeps variance consistent across layers.
  • ▸
    He initialization: W ~ N(0, 2/n[l-1]). Designed for ReLU activations. Accounts for the fact that ReLU zeroes out half its inputs on average.
Initialization Formulas
Xavier: W[l] = np.random.randn(n[l], n[l-1]) × √(1/n[l-1])
He: W[l] = np.random.randn(n[l], n[l-1]) × √(2/n[l-1])
Exam Point — Hyperparameter Priority Order

When tuning, prioritize in this order: (1) Learning rate α — most impactful. (2) Mini-batch size. (3) Number of hidden units. (4) Number of layers. (5) Learning rate decay. (6) β₁, β₂, ε (only for Adam). Getting α right first is the single most important step.

Section 3

Hyperparameter Tuning Strategies

3.1 Grid Search

  • ▸
    Idea: Define a grid of hyperparameter values (e.g., α ∈ {0.01, 0.001}, layers ∈ {2, 3, 4}) and train a model for every combination.
  • ▸
    Pros: Exhaustive — guaranteed to find the best combination within the grid. Simple to implement.
  • ▸
    Cons: Scales exponentially with number of hyperparameters. 5 values × 5 hyperparameters = 5⁵ = 3125 training runs. Impractical for deep learning.
  • ▸
    Use when: Only 1–2 hyperparameters need tuning and each training run is fast.

3.2 Random Search

  • ▸
    Idea: Randomly sample hyperparameter combinations from defined distributions (not a fixed grid).
  • ▸
    Why better than grid search: In practice, some hyperparameters matter much more than others. Random search covers more distinct values of the important hyperparameters with the same number of trials. (Bergstra & Bengio, 2012)
  • ▸
    Search on log scale: For learning rate, sample from log-uniform distribution: α = 10^(-r) where r ~ Uniform(-4, -1), giving values between 0.0001 and 0.1.
  • ▸
    Use when: More than 2 hyperparameters. This is the standard practice in deep learning.

3.3 Bayesian Optimization

  • ▸
    Idea: Build a probabilistic model (usually a Gaussian Process) of how hyperparameters relate to performance. Use this model to intelligently choose the next set of hyperparameters to try — focusing on regions likely to give improvement.
  • ▸
    Pros: Much more sample-efficient than random search. Each trial informs the next.
  • ▸
    Tools: Optuna, Hyperopt, Weights & Biases Sweeps, Google Vizier.
  • ▸
    Use when: Each training run is expensive and you have a limited budget of trials.

3.4 Coarse-to-Fine Tuning

  • ▸
    Phase 1 — Coarse search: Use random search over a wide range. Run many short experiments (few epochs) to quickly identify which regions of hyperparameter space are promising.
  • ▸
    Phase 2 — Fine search: Zoom into the promising region identified in Phase 1. Run longer experiments with finer resolution.
  • ▸
    Why efficient: Avoids spending compute on clearly bad configurations. Most hyperparameter tuning in industry follows this approach.

3.5 Train / Dev / Test Split

  • ▸
    Training set: Used to update parameters (weights) via backpropagation. The network sees this data during training.
  • ▸
    Dev / Validation set: Used to evaluate different hyperparameter configurations and guide tuning. Never used for backpropagation.
  • ▸
    Test set: Used only once at the end to report final model performance. Gives an unbiased estimate of real-world performance.
  • ▸
    Typical splits: For small datasets (few thousand): 60/20/20. For large datasets (millions): 98/1/1 is common — you have more than enough data for a reliable dev/test estimate.
Common Mistake — Dev Set Leakage

Never tune hyperparameters using the test set. If you use the test set to guide tuning, you are effectively training on it and your final accuracy will be optimistically biased. Always tune using the dev set, and touch the test set only once.

Section 4

Problems with Vanilla Gradient Descent

Before understanding Adam and RMSProp, we must understand why plain gradient descent is insufficient for training deep networks efficiently. These problems motivated the development of all modern optimizers.

4.1 Slow Learning Due to Uniform Learning Rate

  • ▸
    Problem: Vanilla GD uses the same learning rate α for all parameters. But different parameters may need different step sizes — some gradients are large (need small α), some are tiny (need large α).
  • ▸
    Consequence: A single α that works for one parameter may cause oscillations in another.

4.2 Oscillation in Narrow Ravines

  • ▸
    Problem: Many loss surfaces look like elongated valleys (narrow in one direction, long in another). GD oscillates back and forth across the narrow direction while making slow progress along the valley toward the minimum.
  • ▸
    Consequence: High learning rate → oscillates across narrow dimension. Low learning rate → too slow in the long dimension. No α satisfies both.

  │  ↗         ↘
  │            ↗↘↗↘→
  │                 optimal
  └───────────────────── w₁  (long direction, slow progress)

  GD path: zigzags inefficiently instead of going straight
Figure 4.1 — Gradient descent oscillating in a narrow ravine instead of progressing efficiently.

4.3 Noisy Gradients (Mini-batch)

  • ▸
    Problem: Mini-batch gradients are noisy estimates of the true gradient. Each update step can point in a slightly wrong direction.
  • ▸
    Consequence: Even near the minimum, the loss doesn't settle — it keeps oscillating around the minimum due to gradient noise.

4.4 Saddle Points and Flat Regions

  • ▸
    Saddle points: Regions where gradient is near zero but it is not a minimum. GD stalls here because ∂J/∂W ≈ 0 → nearly zero update.
  • ▸
    Plateaus: Flat regions of the loss surface where gradients are tiny. Training slows to a crawl even if still far from the minimum.
Motivation for Advanced Optimizers

All of the problems above — oscillation, uniform step size, noisy gradients, saddle points — motivated the development of more sophisticated optimizers. Momentum addresses oscillation. RMSProp addresses the uniform learning rate problem. Adam combines both, making it the most widely used optimizer today.

Section 5

Gradient Descent with Momentum

5.1 Core Idea

Momentum is inspired by physics. Imagine rolling a ball down a hill — it accelerates in consistent directions and dampens oscillations. Momentum does the same for gradient descent by maintaining a running average (exponentially weighted) of past gradients.

Intuition — Ball Rolling Down a Hill

A ball gains momentum as it rolls. In the narrow ravine scenario: across the narrow direction, gradients oscillate +/− and cancel out in the average (slowing oscillation). Along the long direction, gradients consistently point the same way and accumulate (speeding up progress). Result: faster, smoother convergence.

5.2 Exponentially Weighted Moving Average (EWMA)

Momentum relies on EWMA, which is used extensively in all modern optimizers. It is worth understanding deeply.

Exponentially Weighted Moving Average
Vₜ = β · Vₜ₋₁ + (1 − β) · θₜ
  • ▸
    Vₜ — The smoothed (moving average) value at time step t.
  • ▸
    θₜ — The current raw value (e.g. current gradient).
  • ▸
    β — The decay/momentum coefficient. Controls how much history is remembered. Typical value: 0.9.
  • ▸
    Effective window: The EWMA approximates an average over the last 1/(1−β) values. With β=0.9, it averages over ≈ 10 past values. With β=0.99, ≈ 100 values.
  • ▸
    Bias correction: Early values of V are biased toward 0 (since V₀=0). Corrected as: V̂ₜ = Vₜ / (1 − βᵗ). Important in the first few iterations.

5.3 Momentum Algorithm

Algorithm: Gradient Descent with Momentum
Initialize: V_dW = 0, V_db = 0

On each iteration t:
  Compute dW, db ← via backpropagation

  V_dW = β · V_dW + (1−β) · dW ← momentum for weights
  V_db = β · V_db + (1−β) · db ← momentum for biases

  W := W − α · V_dW ← smooth update
  b := b − α · V_db
  • ▸
    β = 0.9 is the standard default. Rarely needs tuning.
  • ▸
    Effect on oscillating direction: V_dW averages out the positive and negative gradients → oscillation is dampened.
  • ▸
    Effect on consistent direction: V_dW accumulates gradients that consistently point the same way → larger, faster steps in that direction.
  • ▸
    Two hyperparameters: α (learning rate) and β (momentum). α matters more; β=0.9 usually works well.
Section 6

RMSProp — Root Mean Square Propagation

6.1 Core Idea

RMSProp (Hinton, 2012) solves the oscillation problem from a different angle than Momentum. Instead of smoothing the gradient itself, it adapts the learning rate for each parameter individually — dividing by the square root of the recent average of squared gradients.

Intuition

Parameters with consistently large gradients (oscillating direction) get a smaller effective learning rate — this slows them down and reduces oscillation. Parameters with consistently small gradients (valley direction) get a larger effective learning rate — this speeds them up. Each parameter adapts its own step size.

6.2 RMSProp Algorithm

Algorithm: RMSProp
Initialize: S_dW = 0, S_db = 0

On each iteration t:
  Compute dW, db ← via backpropagation

  S_dW = β₂ · S_dW + (1−β₂) · dW² ← EWMA of squared gradients
  S_db = β₂ · S_db + (1−β₂) · db²

  W := W − α · dW / (√S_dW + ε) ← adaptive learning rate
  b := b − α · db / (√S_db + ε)
  • ▸
    dW² — Element-wise squaring of the gradient. This removes sign information — we only care about magnitude of recent gradients.
  • ▸
    S_dW — EWMA of squared gradients. Large when gradients have been consistently large; small when gradients have been small.
  • ▸
    Division by √S_dW — Large S_dW → small step (slow down). Small S_dW → large step (speed up). This is the adaptive learning rate.
  • ▸
    ε (epsilon) — A tiny constant (e.g. 10⁻⁸) added to the denominator to prevent division by zero when S_dW ≈ 0.
  • ▸
    β₂ — Controls how fast the squared gradient history decays. Typical value: 0.999.
  • ▸
    α is still a hyperparameter in RMSProp — it controls the base step size. But it can be set larger than vanilla GD because RMSProp itself controls oscillation.

6.3 RMSProp on the Narrow Ravine

  Narrow Ravine — Gradient behavior:

  Direction b (narrow, oscillating):   db is LARGE each step
    → S_db is LARGE
    → α / √S_db is SMALL
    → Small steps in b direction  →  Less oscillation ✓

  Direction W (long, slow progress):   dW is SMALL each step
    → S_dW is SMALL
    → α / √S_dW is LARGE
    → Large steps in W direction  →  Faster progress ✓
Figure 6.1 — How RMSProp adapts step sizes to reduce oscillation and speed up convergence.
RMSProp vs Momentum — Key Difference

Momentum smooths the gradient using EWMA of the gradients themselves (addresses noisy direction). RMSProp divides by the EWMA of squared gradients (addresses scale of gradients). They attack the same convergence problem from different angles — which is why combining them (Adam) works so well.

Section 7

Adam Optimizer — Adaptive Moment Estimation

7.1 What is Adam?

Adam (Kingma & Ba, 2014) is the most widely used optimizer in deep learning today. It combines the best of Momentum and RMSProp: it maintains a smoothed gradient (like Momentum) AND adapts the learning rate per parameter (like RMSProp). It also includes bias correction, making it work well from the very first iteration.

Adam = Momentum + RMSProp + Bias Correction

Momentum tracks the 1st moment (mean) of gradients. RMSProp tracks the 2nd moment (uncentered variance). Adam tracks both, and corrects for initialization bias. Hence the name: Adaptive Moment Estimation.

7.2 Adam Algorithm — Complete

Algorithm: Adam (Adaptive Moment Estimation)
Initialize: V_dW = 0, V_db = 0 ← 1st moment (momentum)
S_dW = 0, S_db = 0 ← 2nd moment (RMSProp)
t = 0 ← iteration counter

On each iteration:
  t = t + 1
  Compute dW, db ← via backpropagation

  Step 1: Update 1st moment (momentum)
  V_dW = β₁ · V_dW + (1−β₁) · dW
  V_db = β₁ · V_db + (1−β₁) · db

  Step 2: Update 2nd moment (RMSProp)
  S_dW = β₂ · S_dW + (1−β₂) · dW²
  S_db = β₂ · S_db + (1−β₂) · db²

  Step 3: Bias correction
  V̂_dW = V_dW / (1 − β₁ᵗ)
  V̂_db = V_db / (1 − β₁ᵗ)
  Ŝ_dW = S_dW / (1 − β₂ᵗ)
  Ŝ_db = S_db / (1 − β₂ᵗ)

  Step 4: Parameter update
  W := W − α · V̂_dW / (√Ŝ_dW + ε)
  b := b − α · V̂_db / (√Ŝ_db + ε)

7.3 Adam Hyperparameters — Explained

HyperparameterRoleDefault ValueNotes
α Learning rate — base step size 0.001 The main hyperparameter to tune. Typical range: 1e-4 to 1e-2
β₁ Decay rate for 1st moment (momentum) 0.9 Controls how much recent gradient is smoothed. Rarely changed.
β₂ Decay rate for 2nd moment (squared gradient) 0.999 Smooths the squared gradient. Almost never changed in practice.
ε Numerical stability constant 10⁻⁸ Prevents division by zero. Almost never changed.
Practical Note — Adam in Industry

In practice, β₁=0.9, β₂=0.999, ε=10⁻⁸ are almost always left at their defaults. The only hyperparameter typically tuned in Adam is α. This is one of Adam's major advantages — it requires very little tuning compared to SGD with Momentum.

7.4 Why Bias Correction Matters

  • ▸
    Problem: V_dW and S_dW are initialized to 0. In early iterations, they are biased toward zero — they underestimate the true first and second moments.
  • ▸
    Example without correction: At t=1, β₁=0.9: V_dW = 0.9×0 + 0.1×dW = 0.1×dW. This is 10 times smaller than the actual gradient! The first few steps would be tiny.
  • ▸
    Correction: V̂_dW = V_dW / (1−β₁ᵗ). At t=1: divisor = 1−0.9¹ = 0.1, so V̂_dW = 0.1×dW / 0.1 = dW. Correct! As t→∞, the divisor → 1 and correction has no effect.

7.5 Adam's Advantages — Why It Dominates

  • ▸
    Adaptive per-parameter learning rates: Each weight gets its own effective α — faster for infrequently updated weights, slower for frequently updated ones.
  • ▸
    Works well in sparse gradients: Useful in NLP/embeddings where many parameters see zero gradients most of the time.
  • ▸
    Robust to noisy gradients: Both moment estimates smooth out noise.
  • ▸
    Fast initial convergence: Bias correction ensures large, correct steps from the very beginning.
  • ▸
    Works across domains: Same default hyperparameters work reasonably well for images, NLP, audio, tabular data — almost no tuning needed.

7.6 Limitations of Adam

  • ▸
    Generalization gap: Models trained with Adam sometimes generalize slightly worse than those trained with SGD + Momentum (Wilson et al., 2017). SGD with carefully tuned α finds "flatter" minima that generalize better.
  • ▸
    Memory overhead: Stores V_dW and S_dW for every parameter — doubles or triples memory compared to SGD.
  • ▸
    AdamW fix: Standard Adam does not correctly implement L2 regularization (weight decay). AdamW (Decoupled Weight Decay) fixes this and is the preferred version in modern practice (used in BERT, GPT, Vision Transformers).
Exam Point — Adam Update Equation (Must Know)

The final Adam update step combines both corrected moments:

W := W − α · V̂_dW / (√Ŝ_dW + ε)

Numerator V̂_dW: smoothed gradient direction (from Momentum). Denominator √Ŝ_dW: scale of recent gradients (from RMSProp). Together: take a smooth step, but scale it appropriately per parameter.

8.1 Full Comparison Table

8.2 Which Optimizer to Use When?

Default choice

Adam (or AdamW). Works out of the box for almost all deep learning tasks. Start here.

Need best generalization

SGD + Momentum with learning rate schedule. Takes more tuning but often best test accuracy for image models (CNNs).

NLP / Transformers

AdamW with linear warmup + cosine decay. This is the standard for BERT, GPT, and all modern language models.

RNNs / LSTMs

RMSProp or Adam. Handles the sparse and variable gradient magnitudes common in recurrent networks.

Section 9

Domain Knowledge in Deep Learning

9.1 What is Domain Knowledge?

Domain knowledge refers to expert understanding of the specific field or problem area in which a deep learning model is being applied — such as medicine, finance, speech processing, biology, or computer vision. It is the accumulated human expertise about how the data was generated, what patterns are meaningful, what features matter, and what constraints must hold.

Core Idea

Deep learning models learn from data. But data alone rarely contains all the information needed to build a great model. Domain knowledge fills the gaps — it guides how we collect data, design architectures, choose features, interpret results, and ensure the model is trustworthy in its deployment context.

9.2 Role of Domain Knowledge — Where it Applies

A. Data Collection and Curation

  • ▸
    What data to collect: A cardiologist knows which ECG features indicate arrhythmia — guiding which signals to record, at what sampling rate, with what labeling criteria.
  • ▸
    Label quality: Medical labels must come from qualified clinicians. Financial labels must account for look-ahead bias. Domain experts define what the "ground truth" label actually means.
  • ▸
    Data imbalance awareness: In fraud detection, fraudulent transactions are <1% of data. Domain knowledge tells you this class imbalance exists and guides the choice of sampling strategy and evaluation metric.

B. Feature Engineering

  • ▸
    Informative features: A clinician knows that the ratio of two blood markers is more informative than the individual values — a raw model would never discover this with limited data.
  • ▸
    Feature normalization: In finance, return normalization removes market regime effects. In biology, log-transforming gene expression data makes it closer to normally distributed.
  • ▸
    Data augmentation strategy: In medical imaging, horizontal flipping of a chest X-ray is valid augmentation (left-right symmetric). But flipping up-down is not — anatomy has a clear up/down orientation. Only a domain expert knows which augmentations preserve label validity.

C. Architecture Design

  • ▸
    Spatial locality in images: Knowledge that nearby pixels are correlated → Convolutional Neural Networks (CNNs) with local receptive fields. This architectural prior dramatically improves efficiency.
  • ▸
    Temporal sequences: Knowledge that speech and text are sequential → RNNs, LSTMs, Transformers. A feedforward network would discard temporal ordering.
  • ▸
    Symmetry constraints: In molecular property prediction, a molecule looks the same regardless of how we rotate it in 3D. Domain knowledge of this symmetry → equivariant graph neural networks that respect rotational symmetry.
  • ▸
    Multi-scale structure: Medical images have relevant patterns at multiple scales (whole organ → tissue → cell). Domain knowledge → multi-scale architectures like U-Net.

D. Loss Function Design

  • ▸
    Asymmetric costs: In medical diagnosis, missing a cancer (false negative) is far worse than a false positive. Domain knowledge → use a weighted loss that penalizes false negatives more heavily.
  • ▸
    Physics constraints: In fluid dynamics simulation, the model output must satisfy conservation of mass. Domain knowledge → add a physics-based penalty term to the loss (Physics-Informed Neural Networks).
  • ▸
    Metric alignment: In speech synthesis, standard MSE loss does not align with human perception of audio quality. Domain knowledge → use perceptual loss functions (e.g. PESQ, STOI) that match human hearing.

E. Evaluation and Interpretation

  • ▸
    Choosing the right metric: In cancer screening, accuracy is a terrible metric (99% accuracy by predicting all negative). Domain knowledge → use sensitivity, specificity, AUC-ROC, or F1 score.
  • ▸
    Sanity checking predictions: A model predicting negative patient age, or a financial model predicting returns above 1000% daily, is clearly wrong. Domain knowledge lets you catch nonsensical outputs.
  • ▸
    Interpretability requirements: In regulated industries (banking, medicine), knowing the model works is not enough — you must explain why it made a decision. Domain experts define what constitutes a valid explanation.

F. Transfer Learning and Pre-training

  • ▸
    Choosing a base model: Domain knowledge tells you whether ImageNet-pretrained features are relevant for your task. For natural images: highly relevant. For satellite imagery: somewhat relevant. For microscopy slides: less relevant — consider domain-specific pretraining.
  • ▸
    Which layers to freeze: Early layers learn universal features (edges, textures) — these can be frozen. Later layers are task-specific — these should be fine-tuned. Domain knowledge determines how different your task is from the pretraining task.

9.3 Domain Knowledge vs Pure Data-Driven Learning

AspectPure Data-Driven (No Domain Knowledge)Domain Knowledge-Guided
Data needed Very large — model must discover everything from data Less — domain priors reduce the hypothesis space
Generalization May learn spurious correlations in the training set Better generalization — inductive biases constrain the model to valid solutions
Architecture Generic (fully connected layers) Task-specific (CNNs, RNNs, GNNs guided by data structure)
Failure modes Hard to detect — model can be confidently wrong in a physically impossible way More constrained — domain constraints prevent some failure modes
Interpretability Black box Predictions can be validated against domain understanding

9.4 Importance in Specific Domains

Healthcare

Anatomical knowledge shapes architectures (U-Net for organs). Clinical knowledge defines valid labels and crucial evaluation metrics (sensitivity, specificity).

NLP

Linguistic knowledge (syntax, semantics, coreference) informed early architectures. Pre-training on language modeling reflects the domain prior: language has long-range dependencies.

Finance

Market microstructure knowledge guides feature design. Regulatory constraints define what decisions the model is allowed to make. Look-ahead bias must be carefully avoided.

Physics / Science

Physical laws (conservation, symmetry) are encoded as constraints. Physics-Informed Neural Networks (PINNs) embed PDEs directly into the loss function.

9.5 The Right Balance — Domain Knowledge + Data

  • ▸
    Too little domain knowledge: The model must learn everything from scratch. Requires enormous data and compute. May learn wrong patterns. Hard to debug.
  • ▸
    Too much domain knowledge (over-constrained): The model cannot capture patterns that violate our assumptions. Misses genuinely novel discoveries in the data.
  • ▸
    Right balance: Use domain knowledge to define the search space and architecture, but let the data fit the parameters within that space. This is the philosophy behind CNNs, RNNs, GNNs, and Transformers — they encode structural priors but learn their weights from data.
Exam Point — Why Domain Knowledge Matters in Deep Learning

1. Inductive bias: Domain knowledge encodes assumptions about the data structure that make learning more efficient. CNNs assume spatial locality. RNNs assume temporal structure. These biases allow learning from less data.

2. Data efficiency: With domain-guided feature engineering, a smaller model can outperform a larger purely data-driven model on limited data.

3. Trustworthiness: In high-stakes applications (medicine, law, safety), a model that incorporates domain knowledge can be more easily validated, audited, and trusted by practitioners.

4. Problem formulation: Choosing the right input, output, loss function, and evaluation metric is impossible without domain understanding. Getting this wrong means even a perfect optimizer cannot save the model.

Module 2 — Complete Summary

  • 1.
    Hyperparameters are set before training. Parameters are learned during training. Hyperparameters include α, number of layers, batch size, epochs, regularization.
  • 2.
    Learning rate α is the most important hyperparameter. Tune it first, on a log scale. Use learning rate schedules for better convergence.
  • 3.
    Tuning strategies: Random search is better than grid search for deep learning. Coarse-to-fine and Bayesian optimization are used in practice. Always tune on the dev set, not the test set.
  • 4.
    Vanilla GD problems: oscillation in ravines, uniform α for all params, noisy gradients, saddle points. These motivated Momentum, RMSProp, and Adam.
  • 5.
    Momentum: Uses EWMA of gradients (β₁=0.9) to smooth the update path and accelerate learning in consistent directions.
  • 6.
    RMSProp: Divides each gradient by √(EWMA of squared gradients) to adapt step size per parameter. Large gradient history → smaller step.
  • 7.
    Adam = Momentum + RMSProp + Bias Correction. Default optimizer in deep learning. Only α needs tuning. Update: W := W − α · V̂_dW / (√Ŝ_dW + ε).
  • 8.
    Domain knowledge guides data collection, feature engineering, architecture design, loss function choice, and evaluation. It encodes inductive biases that improve data efficiency, generalization, and trustworthiness.
